# 诊断页改造计划

## 页面定位
- 页面对应：`frontend/src/views/DiagnosisView.vue`
- 页面职责：用尽量低的认知负担完成快速诊断，为后续规划页提供判断依据
- 设计目标：轻松、简单、快速完成基本评测

## 当前判断
- 当前诊断页结构清楚，单页问答形式已经基本符合任务要求
- 主要问题不是缺功能，而是还能进一步提升“快速完成”的节奏感与稳定层级

## 改造目标
- 保持轻量，不把诊断页做成复杂问卷
- 更明确地告诉用户“为什么要做这一步”
- 让三道题像一个短流程，而不是三个松散表单块

## 计划调整

### 1. 首屏引导再收敛
- 标题、副标题、主题卡保留
- 把介绍文案压缩成一句“我们先判断你当前卡点，再决定从哪一步开始”
- 降低解释语气，提升动作感

### 2. 题目层级优化
- 继续保持 3 题结构，不增加题数
- 每题卡片统一为“题号 + 问题 + 单选项”，减少卡片之间的视觉差异
- 强化当前已选项状态，让填写过程更顺滑

### 3. 提交区优化
- 提交按钮前增加极轻量进度提示，例如“还差 1 题”
- 提交按钮文案更明确指向下一步，例如“生成学习规划”
- 防止用户提交前不清楚下一页会发生什么

### 4. 诊断到规划的衔接
- 提交后的过渡文案要更像“正在根据你的卡点生成学习决策”
- 强化诊断页与规划页之间的因果关系，为规划页的“决策感”做铺垫

## 实施顺序
1. 收紧 `DiagnosisView.vue` 页头文案
2. 统一三道题卡片的视觉节奏和选中反馈
3. 在提交区补充最小化进度提示
4. 微调跳转过渡文案，突出“诊断 -> 规划”的判断链

## 验收标准
- 用户能在很短时间内理解这页只需要完成 3 个选择
- 页面没有多余解释，不会产生“答问卷压力”
- 提交动作清楚对应到“生成规划”
- 整页更像学习流程的诊断环节，而不是普通表单页

## 备注
- 诊断页仍应保持轻量微调，不建议引入新的复杂交互
- 优先复用现有 `WorkflowPageScaffold`、`FormCard` 和按钮体系
