# STRUCTURE 页面规格书（DFS / BFS · 题型判断骨架）

适用范围：任务执行页 · 第一阶段 STRUCTURE · 知识点限定为数据结构中的 DFS 与 BFS。  
读者：产品经理、学习科学设计、交互设计、前端开发。

---

## 1. 页面一句话定义

本页是「DFS/BFS 题型判断骨架」的补全工作台：用户在一套固定判断维度上做出选择或归类，把空骨架填满，形成可复用的题型识别模板，而不是做题、聊天或背概念。

---

## 2. 页面核心目标

帮助用户建立**可操作的 DFS/BFS 题型判断骨架**：给定典型题目表述时，能快速对齐「图模型 / 遍历语义 / 约束与边界 / 输出形态」四类判断，并据此在 DFS 与 BFS 之间做**结构化**取舍（先判题型结构，再谈算法细节）。

---

## 3. 本阶段学习目标

完成本页后，用户能够：

- 从题目中**剥离**出：是否隐含图或状态空间、边与方向性、起点与可达性要求。
- 用同一套维度区分：**深度优先语义**（路径、回溯、穷尽分支）与 **广度优先语义**（最短步数、层序、最近层）。
- 在脑中形成**固定检查顺序**：模型 → 遍历目标 → 约束 → 答案形式，再映射到 DFS 或 BFS。
- **不**要求在本页熟练编码实现；**不要求**在本页完成证明或大量练习。

---

## 4. 用户在本页的唯一主动作

**补全 DFS/BFS 判断骨架**：在骨架工作区内，对每个判断节点完成选择、拖拽归类或点选标签（实现形态由交互设计选定一种主交互，见第 8、9 节），直至骨架无空缺且可提交。

---

## 5. DFS/BFS 结构骨架定义（判断维度）

骨架由以下**固定维度**组成，顺序不可打乱（自上而下或从左到右一致即可，全站统一）：

| 序号 | 维度名称 | 用户要回答的问题（内部用语可再压缩为标签） |
|------|----------|--------------------------------------------|
| D1 | **问题模型** | 这是图上的路径/连通问题，还是状态空间的转移问题？（二选一或「图 / 状态空间」标签） |
| D2 | **遍历目标** | 核心是「找一条路径 / 试所有可能 / 最短步数或最少边」中的哪一种？（单选，与 D1 联动可选灰显非法项） |
| D3 | **约束与边界** | 是否需要避免重复访问、是否有层数/深度上限、是否多源、是否有向/无权等（用预设标签多选，上限如 4 条，避免长文） |
| D4 | **输出形态** | 答案要的是「具体路径 / 步数或层数 / 是否可达 / 所有解」中的哪一种？（单选） |
| D5 | **DFS/BFS 倾向** | 根据 D1–D4 的组合，当前题更偏 **DFS** 还是 **BFS**？（单选，须在 D1–D4 已填后解锁或允许「待判定」占位） |

**骨架逻辑关系（供开发与校验用，不必全文展示给用户）：**

- D2 为「最短步数 / 最少边 / 层序」类 → 倾向 **BFS**（在无权最短路等典型设定下）。
- D2 为「所有路径 / 回溯 / 穷尽」类 → 倾向 **DFS**。
- D5 必须与 D1–D4 的组合**一致**方可判为通过；不一致时给**指向性**纠错（见第 9 节），不展开算法课。

## 8. 每个模块的目标、展示内容、交互方式

### M1 顶栏：阶段与上下文

- **目标**：标明当前处于 STRUCTURE，且本轮是「判断骨架」而非其他阶段。  
- **展示**：阶段名「结构建立」+ 副标「DFS/BFS」或「题型判断骨架」（二选一，全站统一）。  
- **交互**：无；或可点击「？」展开 1 屏内极简说明（可选，默认收起）。

### M2 主任务与完成标准

- **目标**：1 秒内传达「现在要做什么、怎样算做完」。  
- **展示**：主标题「补全 DFS/BFS 判断骨架」；副文一行「完成标准：填齐四维，第五维与之一致」。  
- **交互**：无。

### M3 题目锚点（只读）

- **目标**：给骨架一个具体依附对象，避免抽象空谈；**不**承担教学长篇。  
- **展示**：一道**短**题干或题目标签（如「迷宫最短路」「全排列」「岛屿数量」类的一句话场景），字数上限建议 80 字内；可切换「题例」在 2～3 个预设间切换（若做得到）。  
- **交互**：仅切换题例或展开/收起一行（可选）；**禁止**在此输入长文。

### M4 骨架工作区

- **目标**：承载第 5 节全部维度，是用户唯一高频操作区。  
- **展示**：D1～D5 的节点与占位（见第 9 节）。  
- **交互**：点选、单选组、多选（带上限）、或拖拽标签入位；**一种**为主，避免混用多种复杂手势。

### M5 提示区

- **目标**：在卡壳时给**方向**，不替代用户判断。  
- **展示**：最多 2 条短句，例如「先看要最短还是全部」「先定是图还是状态」；可随 D1～D4 空缺状态变化。  
- **交互**：可手动展开；或首次进入展开，之后折叠。

### M6 主操作：提交 / 校验

- **目标**：完成骨架后的明确收尾动作。  
- **展示**：主按钮「提交判断」；未填满时禁用或二次确认（产品二选一）。  
- **交互**：点击触发校验（见第 9、10 节）。

### M7 反馈条

- **目标**：告知是否通过，以及**下一步**（进入 UNDERSTANDING 或修正骨架）。  
- **展示**：一行结果 + 一条动作指引；错误时不堆叠长列表。  
- **交互**：通过态可出现「继续」；未通过「改骨架」聚焦到首个冲突维度。

### M8 阶段进度

- **目标**：保持工作流感，不抢骨架视觉。  
- **展示**：STRUCTURE 高亮，其他阶段灰显或点状。  
- **交互**：通常不可点或仅查看。

## 9. 骨架工作区详细规格

### 9.1 节点

- **节点**：D1～D5 各为一个**卡片节点**，视觉同级；D5 可略强调为「结论」但不大于 D1～D4 合计面积。  
- **依赖**：D5 在交互上建议在 D1～D4 均有值后才可提交；或允许 D5 为「待选」但与提交禁用联动。

### 9.2 占位

- **未填**：显示「待选」或虚线占位，**禁止**显示长篇占位符文案。  
- **已填**：显示用户所选**标签文本**（短词，如「状态空间」「最短步数」）。  
- **冲突**（提交后校验失败）：对应节点边框高亮，**一条**纠错短句（见 9.4）。

### 9.3 用户操作方式（需统一为一种主方案）

**方案 A（推荐比赛演示）**：每节点内**单选/多选 chips**，点选即填入。  
**方案 B**：右侧「标签池」拖拽到各节点槽位。  
**方案 C**：下拉 + 标签组合（认知负担略高，慎用）。

无论哪种：**禁止**要求用户输入自由文本完成骨架。

### 9.4 正确反馈方式

- **即时反馈（可选）**：仅可做「已选」态，**不做**对错判直到提交（降低测验感）；若做即时合法性提示，仅限制「互斥选项」（如 D2 与 D4 明显矛盾）用轻提示，不剧透答案。  
- **提交后**：  
  - **全对**：骨架整体变为「已通过」态（如描边或浅底），M7 显示通过文案 + 「继续」。  
  - **部分错**：标红**第一个**逻辑冲突节点（按 D1→D5 顺序），一句说明如「最短步数通常对应另一种遍历」+ 不透露完整标准答案树。  
  - **多错**：仍优先只强调**一处**，避免清单式打击。

### 9.5 答案与规则来源

- 判定规则由产品/教研给定**确定性规则表**（D1–D4 组合 → 期望 D5），**不依赖** LLM 实时裁决核心对错；LLM 仅可润色短反馈用语（见第 11 节）。

## 10. 页面状态流转

| 状态 | 条件 | 界面表现 | 用户可执行 |
|------|------|----------|------------|
| **初始态** | 进入本页，题例已加载 | 骨架全空或部分默认值；提交禁用或可用但点后有校验 | 填充 D1～D5 |
| **操作中** | 任一节点有交互 | 占位减少；提示可随空缺变化；无聊天流 | 继续填充或修改 |
| **提交后 · 未通过** | 校验失败 | 冲突节点高亮 + 一条纠错；其余保持用户选项 | 修改后再次提交 |
| **提交后 · 通过** | 校验成功 | 骨架锁定或只读通过态；主任务区出现「继续」或系统自动进入下一阶段（依全局流程） | 进入下一阶段或回看只读骨架 |

**不允许**的状态：无限制的「闲聊中」、无骨架中心的「做题得分中」、长表单「填表提交中」作为主状态名。

## 11. AI 导师在本页的定位、权限、出现方式

- **定位**：辅助**措辞**与**极短提示**，不接管骨架结构、不替代规则判定。  
- **权限**：  
  - **可**：在提示区生成或润色 ≤2 条短提示；在用户提交错误后，用不同措辞重复「指向性」纠错（仍须受规则表约束，不编造判题逻辑）。  
  - **不可**：展开长篇讲解、给出完整解题代码、与用户多轮对话主导页面、改变 D1～D5 定义或顺序、直接告诉「选第几个选项」式代答（若产品严格要求防代答，则错误反馈也只描述矛盾类型）。  
- **出现方式**：  
  - 默认**无对话框**；可选「问一句」入口，点开为**单轮**短答（如仅澄清题干关键词），答完收起。  
  - 不在首屏自动弹出大段欢迎语。

---

## 12. 文案风格要求

- **克制**：每模块标题 ≤12 字为宜；说明句单行优先。  
- **短句**：动词开头说明动作，如「补全骨架」「选一个模型」「提交判断」。  
- **只说明要做什么**：不写「本阶段旨在培养……」；不写系统原理长文。  
- **错误提示**：指出**改哪里**或**矛盾类型**，不写论文式解释。  
- **术语**：与产品统一——STRUCTURE 显示为「结构建立」；DFS/BFS 用中文或英文一套，全页一致。

---

## 13. 禁止项（不能出现的 UI 和交互）

- 聊天列表、消息气泡流、多轮对话作为主界面。  
- 选择题测验卷样式（题号+ABCD 连篇、无骨架结构）作为主形态。  
- 长表单、多页表单项、大量自由文本输入完成本阶段任务。  
- 首屏多个并列主按钮或多个「主任务」标题。  
- 大段 Markdown/讲义阅读区占据中心。  
- 以「输入你的理解」为主输入框且比骨架更显眼。  
- 骨架未完成时即给最终算法步骤或代码作为中心内容。  
- 动画炫技干扰判断（轻微状态反馈除外）。

---

## 14. 为什么这个页面适合比赛展示

- **产品立场清晰**：一眼可见「不是聊天软件」，而是**规则 + 脚手架**的学习工作台。  
- **可演示、可截图**：骨架补全过程短、结果可视化强，评委易理解**结构化学习**在做什么。  
- **技术可信**：核心判题可走规则引擎，LLM 可有可无，符合「规则为主、LLM 为辅」的叙事。  
- **差异化**：同场常见「AI 问答」或「刷题 App」相比，本页突出**题型判断元能力**与**低负担高引导**的交互设计。
