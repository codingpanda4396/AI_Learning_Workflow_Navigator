# AI Learning Workflow Navigator 项目现状总结

## 1. 项目一句话概括

这是一个围绕“如何更高质量地使用 LLM 学习”而设计的学习流程导航系统，不是通用聊天工具，也不是传统课程平台。  
它试图把用户从“提问拿答案”升级到“目标澄清、状态诊断、路径规划、任务执行、过程反思、结果复盘”的完整学习闭环。

当前项目已经具备比较完整的主流程雏形：

`目标输入 -> 诊断 -> 学习计划预览 -> 计划提交 -> 任务执行 -> 学习报告 -> 下一步决策`

## 2. 业务层面总结

## 2.1 业务核心问题

项目解决的不是“用户问不到答案”，而是“用户不会用 AI 形成有效学习行为”。

从业务意图上看，系统主要想解决以下问题：

- 用户目标通常模糊，不知道该学什么、先学什么。
- 用户对自己当前所处阶段缺少判断，难以准确识别卡点。
- 用户即使使用 LLM，也常停留在直接索要答案，而不是构建理解。
- 学习过程缺少中间反馈，无法动态调整路径。
- 学习结束后缺少结构化沉淀，难以形成成长资产。

## 2.2 当前业务主链路

### 1. 目标输入

用户输入原始学习目标，并补充时间预算、基础水平、偏好入口、学科提示、知识点提示等信息。  
系统会把原始文本结构化，产出：

- 结构化学习目标
- 目标上下文快照

业务意义是把“我想学一下这个”转换成系统可判断、可规划的目标对象。

### 2. 诊断

系统会先创建诊断会话，再通过一组固定问题快速定位用户当前状态。  
诊断结果不只是标签化分类，还会进一步生成：

- 学习者画像快照
- 诊断证据摘要
- 学习策略画像

业务意义是回答“用户现在处在什么状态、主要卡在哪里、更适合怎样切入”。

### 3. 学习规划

系统根据目标和诊断结果进行策略选择，再生成任务化学习计划。  
规划层不是一次性吐出大段建议，而是输出：

- 推荐策略
- 推荐切入点
- 分阶段任务
- 成功标准
- 风险提示
- 关键证据

业务意义是把“建议”转化成“可执行路径”。

### 4. 任务执行

执行阶段是当前项目最有产品辨识度的部分。  
系统不是只展示任务标题，而是把每个任务拆成引导式学习流程，包括：

- 当前任务目标
- 推荐提问模板
- 探索对话
- 自我解释
- 微检查点
- 补救流程
- 收束总结

业务意义是把“学习任务”变成“受约束、可推进、可评估”的行为流程。

### 5. 报告与下一步

任务全部完成后，系统会聚合执行证据，生成学习报告和下一步建议。  
当前报告不仅看是否完成任务，还尝试分析：

- 任务完成度
- 未解决问题
- 证据摘要
- 学习方法表现
- 下一步动作建议

业务意义是让系统从“过程推动器”进一步成为“学习策略调节器”。

## 2.3 当前业务定位判断

从业务设计看，这个项目更接近：

- AI 学习教练
- 学习流程编排器
- 带诊断与复盘能力的任务型学习助手

而不是：

- 通用 AI 问答产品
- 内容生成型刷题工具
- 课程平台或知识库平台

## 2.4 当前业务成熟度判断

当前项目已经不只是 Demo 页面，而是已经形成了较完整的 MVP 业务闭环，尤其在“诊断-规划-执行-复盘”这条链路上已经比较完整。  
其中最成熟的部分是：

- 学习主流程建模
- 执行态任务引导机制
- 结果复盘与下一步决策逻辑

相对还可以继续增强的业务部分是：

- 多轮长期学习资产沉淀
- 用户历史 session 的持续利用
- 更灵活的诊断题库与个性化策略
- 真正长期连续的再规划机制

## 3. 前端技术层面总结

## 3.1 前端技术栈

前端采用的是典型的轻量现代 Web 技术栈：

- Vue 3
- TypeScript
- Vite
- Pinia
- Vue Router
- Tailwind CSS
- Axios

特点是技术栈比较克制，没有引入过重的状态框架或 UI 框架，适合快速迭代业务流程型产品。

## 3.2 前端结构特点

前端整体按“页面流转 + 业务组件 + API 层 + 状态层”组织，结构比较清晰。

主要结构可以概括为：

- `views/` 承担各业务阶段页面
- `components/` 承担页面复用组件、执行引导组件、AI Tutor 组件
- `stores/` 管理工作流状态和 AI Tutor 状态
- `api/` 统一封装后端请求
- `utils/` 承担提交映射、展示模型转换、任务收束 payload 构造等辅助逻辑
- `types/` 承担 DTO、枚举、标签映射

这是比较标准且易维护的分层方式。

## 3.3 前端业务路由设计

路由围绕主流程设计，核心页面包括：

- `/goal`
- `/diagnosis`
- `/plan`
- `/execution`
- `/tasks/:taskId/run`
- `/report`

路由守卫依赖 `workflow store` 中的关键 ID：

- `goalId`
- `diagnosisId`
- `planId`
- `sessionId`
- `currentTaskId`

这意味着前端已经把“业务状态推进”作为路由控制条件，而不是单纯页面跳转，比较符合流程型产品特征。

## 3.4 前端状态管理特点

`frontend/src/stores/workflow.ts` 是当前前端的主业务状态中心，保存了从目标、诊断、计划、会话到报告的整条链路状态。  
同时用 `sessionStorage` 持久化关键 ID，保证页面刷新后流程不立即丢失。

这套设计的优点：

- 业务上下文集中
- 页面切换成本低
- 支持流程恢复

局限也比较明显：

- 关键数据很多仍然依赖前端内存态
- 浏览器刷新后可恢复的主要是 ID，不是完整业务快照
- 多 tab / 多端一致性能力较弱

## 3.5 前端页面能力判断

### 目标页

目标页不是单纯输入框，而是加入了：

- 快速预设场景
- 时间预算
- 基础水平
- 学习偏好
- 主题与知识点提示

说明产品在入口上已经开始考虑降低用户表达门槛。

### 诊断页

诊断页采用“少量固定问题 + 映射提交”的轻量方案，说明当前更强调快速完成状态定位，而不是做重型测评系统。

### 计划页

计划页重展示“当前问题、推荐策略、学习路径、当前阶段”，目标明确，是在帮用户理解系统为什么这样规划。

### 任务执行页

任务执行页是前端最复杂的页面，也是项目的核心交互页面。  
它已经具备：

- 任务引导分步 UI
- 提示模板注入
- 对话式探索
- 自我解释输入
- checkpoint 检查
- 补救提示
- 完成收束表单

说明前端已经不只是渲染后端结果，而是在承接完整的“教学交互状态机”。

### 报告页

报告页承担结果展示和下一步确认，是当前闭环的最后一站。

## 3.6 前端 API 与错误处理

请求层通过 `axios` 统一封装，并适配了后端的 `GlobalResponse` 格式。  
前端会把非 `OK` 结果统一转成业务错误，同时结合错误码映射成人类可读文案。

这是比较成熟的接口消费方式，优点是：

- 页面层逻辑更简单
- 错误提示风格统一
- 前后端契约清晰

## 3.7 前端亮点与问题

### 亮点

- 路由、状态、页面结构都紧贴业务主流程。
- 执行页交互完整，产品辨识度高。
- AI Tutor 被做成浮层和面板，不打断主线流程。
- 展示模型和提交映射被抽到 `utils`，降低了页面组件复杂度。

### 当前问题或潜在问题

- 页面大量直接依赖 store 中的业务上下文，后续复杂度继续上升时可能出现耦合增强。
- 当前流程高度串行，若以后支持历史 session、跳步、回看、多计划并存，前端状态模型需要升级。
- 编码文本存在部分字符显示异常，说明文档或部分文件可能有编码历史问题，需要统一清理。
- 缺少更明显的前端测试痕迹，当前可靠性更多依赖手工联调。

## 4. 后端技术层面总结

## 4.1 后端技术栈

后端采用：

- Java 17
- Spring Boot 3.2
- Spring Web
- Spring WebFlux
- MyBatis-Plus
- PostgreSQL
- Flyway
- Caffeine
- H2（测试）

这说明后端定位是典型的单体应用，但已经兼顾了同步接口、流式输出、数据库持久化与缓存能力。

## 4.2 后端整体分层

后端整体分层比较明确：

- `api`：控制器、DTO、统一响应、异常处理
- `application`：应用服务、规则引擎、规划、执行、报告、LLM orchestration
- `domain`：领域模型、枚举、策略对象
- `infrastructure`：缓存、持久化、LLM client、配置、内存存储

这是偏经典 DDD-lite / 分层架构风格。  
虽然还没有复杂聚合根和严格领域边界，但“控制器薄、应用层重、领域模型独立”的方向是清晰的。

## 4.3 核心后端业务模块

### 目标模块

`GoalApplicationService` 负责：

- 解析原始目标
- 生成结构化目标
- 生成目标上下文快照
- 持久化到 `learning_goal`

说明目标阶段已经不是纯字符串存储，而是经过规则引擎处理的结构化业务对象。

### 诊断模块

`DiagnosisApplicationService` 负责：

- 创建诊断会话
- 生成问题集
- 归一化用户答案
- 生成学习者画像
- 生成诊断证据和策略画像
- 持久化诊断答案、画像快照、学习会话进度

这部分已经具备“规则判断 + 证据输出 + 会话推进”的完整业务职责。

### 规划模块

`PlanningApplicationService` 负责：

- 组装规划上下文
- 选择规划策略
- 生成阶段和任务模板
- 生成推荐切入点
- 预览计划
- 提交计划并物化会话任务
- 为每个任务生成可执行规格 `ExecutableTaskSpec`

这一层本质上是“策略引擎 + 模板工厂 + 任务物化器”。

### 执行模块

`ExecutionApplicationService` 与 `TaskExecutionFlowService` 共同构成执行中枢。

其中：

- `ExecutionApplicationService` 偏会话推进、任务完成、记录聚合
- `TaskExecutionFlowService` 偏任务执行状态机、消息交互、self explanation、checkpoint、runtime 持久化

这部分是项目后端最复杂、也最有含金量的模块。

### 报告模块

`ReportApplicationService` 负责：

- 聚合 session 证据
- 生成学习报告
- 汇总学习方法画像
- 计算下一步动作建议

这表明项目后端并不只是 CRUD，而是已经形成了较强的“行为分析 + 结果决策”逻辑。

## 4.4 后端执行态设计亮点

当前后端最值得关注的是“任务执行态”的实现，它已经明显超出普通表单式任务系统。

### 1. 有显式任务状态机

执行阶段至少覆盖这些状态：

- `ORIENT`
- `EXPLORE`
- `SELF_EXPLAIN`
- `CHECK`
- `REMEDIAL`
- `PASS`

这意味着系统把“学完一个任务”拆成多个认知行为阶段，而不是仅靠一次提交完成。

### 2. 有引导相位

除了执行状态，还叠加了一层指导相位，例如：

- 澄清目标
- 搭框架
- 尝试表达
- 查缺补漏
- 反思
- 过渡提示

这使系统不仅能判断“用户走到哪一步”，还能判断“当前更适合给什么类型的教学引导”。

### 3. 有行为检测

系统会识别用户当前行为类型，例如：

- 求解释
- 求例子
- 求对比
- 求简化
- 自我解释
- 表达困惑
- 直接要答案

这说明任务执行不是纯聊天，而是受行为语义驱动的引导式对话。

### 4. 有证据累积

系统会在执行过程中沉淀：

- runtime 状态
- action history
- task message
- checkpoint result
- task method profile
- completion evidence snapshot

这为报告和下一步决策提供了证据基础。

## 4.5 持久化设计判断

数据库设计已经覆盖了主流程核心对象：

- `learning_goal`
- `learning_session`
- `diagnosis_session`
- `diagnosis_answer`
- `learner_profile_snapshot`
- `learning_plan`
- `session_task`
- `task_interaction`
- `task_completion`
- `task_execution_runtime`
- `task_state_transition`
- `task_message`
- `task_checkpoint_result`
- `task_method_profile`

这说明当前系统已经从“纯内存原型”进化到“关键流程落库”的阶段。

但同时也存在一个过渡特征：

- 运行时仍大量依赖 `InMemoryStore`
- 数据库和内存是并存关系，不是完全以数据库为单一事实源

这通常意味着项目正处于“从原型期走向可持续工程化”的中间阶段。

## 4.6 LLM / AI Tutor 技术实现判断

当前项目有两条和 AI 相关的能力线：

### 1. 页面级 AI Tutor

`AiTutorController` + `AiTutorServiceImpl` 提供：

- prompt
- explain
- feedback
- prefetch
- chat
- chat stream

其特点是：

- 支持普通返回和 SSE 流式返回
- 有 Caffeine 缓存
- 有 fallback 文案兜底
- 支持接入真实 LLM，也支持本地兜底运行

这部分更像“页面伴随式辅导能力”。

### 2. 任务执行中的 Tutor Orchestrator

`TaskTutorOrchestrator` 负责执行阶段对话生成，其特点是：

- 结合任务上下文拼装 prompt
- 根据状态、边界和教学意图控制回复
- 防止直接答案泄露
- LLM 异常时回退到 mock

这部分更像“工作流内嵌的任务导师”。

整体判断是：项目并没有把 LLM 当作唯一核心，而是把 LLM 纳入到受约束的流程编排中，这一点非常关键。

## 4.7 后端亮点与问题

### 亮点

- 分层比较清楚，控制器薄，应用服务职责明确。
- 规划、执行、报告三段业务都已经形成独立模块。
- 执行态状态机设计完整，是项目最核心的技术资产。
- 数据库迁移、持久化实体、Repository 体系已经较齐全。
- LLM 接入有 mock、fallback、cache、streaming 等工程化处理。
- 测试文件数量不少，说明项目对回归验证已有一定重视。

### 当前问题或潜在问题

- `InMemoryStore` 仍承担大量关键运行态，系统重启、一致性、横向扩展能力会受限。
- ID 处理很多依赖字符串前缀和数字提取，后续规模扩大后可维护性一般。
- 应用层服务职责偏重，部分逻辑继续增长后可能需要进一步拆分领域服务或流程编排器。
- 部分模块还保留 `FixedSampleData` 兜底，说明线上真实数据闭环还没有完全彻底化。
- 当前是单体内强流程实现，未来若支持长期学习、跨 session 资产复用，会面临更复杂的数据建模挑战。

## 5. 项目当前阶段判断

综合前端、后端和业务来看，当前项目大致处于：

“MVP 已完成主闭环，正在从可演示产品走向可持续工程化系统”的阶段。

更准确地说，它已经具备：

- 明确的产品定位
- 完整的主业务路径
- 比较强的任务执行中枢
- 初步的 LLM 工程化接入
- 一定程度的持久化与测试基础

但还没有完全进入“稳定生产级平台”阶段，主要差距在于：

- 内存态与数据库双轨并存
- 长周期学习资产沉淀还不够强
- 前后端一致性与扩展性还有升级空间
- 业务规则虽完整，但还偏固定模板和规则驱动

## 6. 总结

如果只看当前实现，这个项目最有价值的地方不是“接了 OpenAI 接口”，而是已经把学习过程拆成了可计算、可引导、可落库、可复盘的流程系统。  
从产品角度，它的核心竞争力在“把 LLM 学习行为流程化”；从技术角度，它的核心资产在“任务执行状态机 + 学习证据沉淀 + 规划/报告闭环”。

如果后续继续演进，最值得重点投入的方向会是：

- 把运行态进一步从内存迁移到稳定持久层
- 强化长期 session / 用户资产复用
- 提升策略与诊断的动态化程度
- 补齐前端和后端更系统的自动化测试
- 继续打磨执行阶段的教学引导质量
