# 二、本轮后端目标

本轮只做最关键的 4 件事，按顺序推进。

## 目标 1：补一个学习反馈报告聚合接口

当前系统已有 practice、submission、feedback、mastery 等能力，但这些能力对前端而言可能仍然偏碎。

你要补一个**面向页面消费的聚合接口**，用于支撑“学习报告/结果导航页”。

### 目标输出语义

接口返回的数据应尽量覆盖：

- 本轮学习/检测总体表现
- 当前节点掌握结论
- 每题评价摘要（或关键题目摘要）
- 薄弱点/易错点
- 推荐下一步动作
- 推荐原因
- 可选：是否已写入成长沉淀/知识图谱

### 实现要求

1. 优先复用现有表与服务，不要另起炉灶。
2. 聚合逻辑放在 application/domain 合理位置，不要堆在 controller。
3. DTO 命名必须页面友好、语义清晰。
4. 如果现有字段不足，做最小必要补充。
5. 不要只把底层记录原样透传给前端。

---

## 目标 2：统一“训练结果 -> 下一步动作”的决策口径

当前系统要更清晰地表达：为什么用户下一步是“继续推进 / 复习 / 强化 / 回退”。

你需要在后端补一层**显式决策逻辑**。

### 要求

1. 先用规则即可，不追求过度智能。
2. 决策结果至少包含：
   - `recommendedAction`
   - `reason`
   - 可选：`targetNodeId` / `targetTaskType` / `confidence`
3. 决策输入优先来自：
   - 练习表现
   - node mastery
   - weak points / diagnosis
4. 规则必须可解释，不允许黑盒魔法。

### 建议规则方向

- 表现好：推进下一知识点
- 表现一般：当前节点补一轮复习/巩固
- 表现差：回退前置知识点或继续强化基础

请把规则封装清楚，便于后续替换为更复杂策略。

---

## 目标 3：补一个最小可用的成长沉淀聚合接口

本轮不要求完整复杂图数据库，但要支持前端做“成长视图 / 个人知识图谱 MVP”。

### 目标输出语义

至少提供其中一类聚合能力：

#### 方案 A：成长仪表盘聚合接口

返回：
- 已学习节点数
- 当前课程/当前方向掌握概览
- 最近训练表现摘要
- Top 薄弱点
- 最近学习报告列表（可选）

#### 方案 B：知识节点掌握度聚合接口

返回：
- 节点列表
- 节点掌握度
- 节点状态（未学 / 学习中 / 待巩固 / 已掌握）
- 当前节点/推荐节点
- 节点间基础关系（如果现有模型支持）

优先选择对现有架构侵入最小，但对前端最有价值的方案；也可以两个都做 MVP。

### 实现要求

1. 复用现有 `concept_node + node_mastery + session/history` 等模型。
2. 如果“节点状态”当前没有现成字段，可通过规则映射生成视图态。
3. 明确区分“底层事实”和“页面聚合视图”。

# 四、实现原则

## 1. 聚合优先

优先给前端一个“能直接渲染页面”的结果，而不是更多底层碎片。

## 2. 单一事实源优先

底层事实依旧来自已有表；不要为了聚合而复制一套业务数据。

## 3. 可解释优先

任何推荐动作都要带解释，不能只有结论。

## 4. 可扩展优先

当前先规则，后续可接更复杂策略，但本轮结构要留扩展点。

## 5. 最小必要改表

如非必要，不要新增复杂表结构。优先利用现有模型做聚合视图。
