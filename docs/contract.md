# contract.md

你是本项目的实现代理（Codex）。你的任务不是自由发挥，而是在**既有产品蓝图、既有代码结构、既有接口语义**之内，做出**可运行、可验证、可回退**的增量实现。

## 0. 项目目标

本项目的目标不是做一个泛化聊天机器人，也不是一个只会生成学习内容的页面集合，而是一个：

**面向大学课程场景的 AI 个性化学习导航系统。**

系统主闭环必须始终围绕下面这条链路展开：

`学习目标输入 -> 能力分析/学习方案 -> 路径规划 -> 分阶段任务学习 -> 个性化检测 -> 自动反馈 -> 下一步学习决策 -> 知识图谱/成长沉淀`

任何实现都必须服务于这个闭环，不能偏离。

---

## 1. 全局硬约束

1. **禁止推翻式重构。**
   - 不要无故重命名大量文件。
   - 不要随意替换现有技术栈。
   - 不要为了“更优雅”而破坏当前能跑的主链路。

2. **只做增量式、可验证修改。**
   - 每次修改必须能解释“为什么改、改了什么、如何验证”。
   - 优先小步快跑，避免一次性改动过大导致系统失控。

3. **前后端契约优先。**
   - 如果涉及新接口或字段，先定义契约，再改实现。
   - 所有新增字段命名必须语义清晰、可长期维护。
   - 避免前端自己猜状态，避免后端返回模糊语义。

4. **产品语义统一优先于技术炫技。**
   - 本项目当前最重要的问题不是“缺少高级算法”，而是“主链路闭环表达要清楚”。
   - 所有实现必须优先增强用户对闭环的感知。

5. **保留 fallback 思维。**
   - 涉及 LLM 的地方，不要把系统做成“没有模型输出就无法工作”。
   - 能规则兜底的地方必须保留兜底。
   - 错误状态必须可解释，不允许黑盒卡死。

6. **不制造新的概念债务。**
   - 不要新增一套和现有 session/task/practice/feedback 平行的模型。
   - 不要引入新的阶段命名体系。
   - 不要出现相同含义多个字段/多个状态来源。

## 2. 统一产品叙事

所有代码、注释、接口、页面文案，尽量向下面的统一叙事收敛：

### 2.1 系统定位

- 不是单点功能集合
- 而是“学习流程导航 + 训练反馈驱动 + 长期成长沉淀”的系统

### 2.2 主业务对象

优先复用并围绕这些对象实现：

- `learning_session`：承载一轮学习会话
- `concept_node`：知识节点/知识结构
- `task`：分阶段学习任务
- `practice_item` / `practice_submission` / `practice_quiz`：检测训练
- `learning_feedback`：学习反馈
- `node_mastery`：掌握度与证据
- `tutor message`：Tutor 辅助对话

### 2.3 统一阶段表达

对外展示尽量统一为以下四阶段：

- 建立框架
- 理解原理
- 训练应用
- 评估反思

说明：
- 内部若已有 `STRUCTURE / UNDERSTANDING / TRAINING / REFLECTION` 或近似枚举，可逐步兼容。
- **不要再引入新阶段名。**
- `REFLECTION` 与 `EVALUATION` 如果并存，优先收敛到统一对外文案。

## 3. 当前阶段的核心开发目标

当前不是继续散着补功能，而是完成下面三件事：

### 3.1 做实训练驱动闭环

用户必须能明显感知：

- 我刚做了什么题
- 系统如何评价我
- 哪些地方薄弱
- 为什么建议我下一步这样学

### 3.2 做出长期成长感

用户必须能看到：

- 我的知识节点在积累
- 我的掌握度在变化
- 系统在记录我的成长轨迹

### 3.3 做出清晰分工边界

- 前端负责把系统做得清楚、可信、可演示
- 后端负责把聚合、决策、状态、统计做扎实

---

## 4. 工程实现约束

### 4.1 修改原则

每次改动都遵守：

1. 先读相关上下文再改
2. 先找现有实现再扩展
3. 优先复用已有 store / service / repository / DTO
4. 保持命名风格一致
5. 保持代码风格一致
6. 改完后补最小必要测试/校验

### 4.2 禁止事项

- 不要无故修改数据库历史迁移文件
- 不要跳过类型定义直接在前端乱用 `any`
- 不要在前端硬编码后端业务结论
- 不要在后端返回“前端自己猜”的原始碎片数据，如果这里本应是聚合视图
- 不要新增与现有页面语义冲突的 route/view
- 不要为了省事复制黏贴出第二套相似逻辑

### 4.3 数据与状态原则

- 单一事实源优先
- 状态流转要能追踪
- 聚合结果优先由后端提供
- 页面只消费明确语义的数据

示例：
- “最近训练结果摘要”“当前薄弱点”“推荐下一步及原因”这类数据，应优先由后端聚合返回，而不是前端拼凑。

### 5.4 后续建议

只给最相关的下一步，不要泛泛而谈。

## 6. 前后端协作约束

### 6.1 如果你在做前端

你必须：

- 先看已有页面与 store 结构
- 优先优化 Session / TaskRun / TaskSubmit / Dashboard 一类核心页面
- 组件设计围绕“主学习区 + 状态摘要 + 下一步引导 + Tutor 辅助”展开
- 文案必须减少术语噪音，增强“用户此刻该做什么”的引导感

你不要：

- 把页面做成信息堆叠墙
- 暴露一堆内部阶段枚举给用户
- 在没有后端契约的情况下编造业务字段

### 6.2 如果你在做后端

你必须：

- 优先补聚合接口，而不是继续增加零散原子接口
- 优先补“反馈报告 / 知识图谱 / 成长统计 / 能力分析 / 学习方案”这类高价值输出
- 保证返回结果语义明确、可解释
- 对状态、规则、兜底有清晰处理

你不要：

- 只返回底层表数据让前端自己拼故事
- 在核心链路中制造多套并行决策逻辑
- 让 LLM 输出直接决定数据库状态而没有校验

## 7. 优先级规则

当需求很多时，按下面优先级决策：

1. 主链路可运行
2. 闭环可感知
3. 前后端契约清晰
4. 数据沉淀合理
5. 视觉和体验增强
6. 额外炫技能力

如果两个方案冲突，优先选择：

- 更稳的
- 更容易验证的
- 更符合当前架构的
- 更有利于竞赛演示的
