# 项目现状总结

## 1. 项目定位

AI Learning Workflow Navigator 当前已经不是单纯的“学习内容生成器”，而是在向一个 MVP 阶段的“AI 学习流程导航系统”收敛。

从业务角度看，项目的核心目标已经比较明确，即支撑这样一条最小闭环：

`学习目标 -> 会话创建 -> 学习路径规划 -> 分阶段任务执行 -> 训练作答 -> 结果反馈 -> 下一步学习建议`

从技术角度看，项目也已经从单一后端骨架，演进为一个前后端联动、具备结构化训练与反馈能力的应用系统，当前包含：

- Spring Boot + PostgreSQL + Flyway 后端
- Vue 3 + Pinia + Vite 前端
- 练习、反馈、Tutor、路径规划等核心模块

## 2. 业务现状

### 2.1 当前已经具备的业务能力

现阶段 MVP 已经具备一条可运行的主业务链路：

1. 用户登录与会话续接
2. 基于课程 / 章节 / 学习目标创建学习会话
3. 为当前会话生成分阶段学习任务
4. 执行不同阶段的任务内容
5. 进入训练阶段，生成或获取练习题并提交答案
6. 产出反馈、薄弱点诊断与掌握度更新
7. 返回下一步行动建议，并保留历史学习记录

这说明项目已经从“展示学习计划”走到了“可交互、可留痕、可反馈”的学习流程原型阶段。

### 2.2 当前业务模型

从现有代码和文档看，当前业务模型已经比较成型：

- 学习载体：`learning_session`
- 知识结构：`concept_node`
- 执行单元：分阶段 `task`
- 训练载体：`practice_item`、`practice_submission`、`practice_quiz`
- 反馈载体：`node_mastery`、`learning_feedback`
- 辅助层：AI Tutor 对话能力

可以看出，项目已经明确采用以下产品思路：

- 用 `session + task` 承载主流程
- 用训练结果作为核心反馈来源
- 用掌握度与薄弱点结果影响“下一步学什么、怎么学”

### 2.3 当前业务优势

- 学习流程已经按阶段拆分，不再是单一大任务，业务语义更清晰。
- 训练与反馈不再停留在概念层，已经有表结构、接口和前端承接页面。
- AI Tutor 已经嵌入任务流，能作为学习辅助层，而不是孤立聊天功能。
- 历史记录和当前会话查询已具备，为后续留存、续学、复盘提供基础。

### 2.4 当前业务短板

目前最大的差距并不是“缺少功能模块”，而是“产品闭环感还不够强”：

- 会话、任务、训练、反馈、Tutor 已经都有，但用户对整条链路的感知仍偏分散。
- 用户做完训练后，训练结果如何影响下一步学习路径，前端层面的呈现还不够直接。
- 术语尚未完全统一，尤其阶段命名上仍存在 `REFLECTION` 与 `EVALUATION` 并存的情况。
- 当前能力分支已经不少，但“训练驱动的个性化学习闭环”这个核心产品叙事还需要进一步收口。

## 3. 技术现状

### 3.1 后端现状

后端基于 Spring Boot 3.4.3、Java 17、PostgreSQL、Flyway 与 Spring JDBC，整体已经形成较完整的分层结构：

- `api`：Controller 与 DTO，负责接口输入输出
- `application`：Service、UseCase、Command、Query，负责业务编排
- `domain`：枚举、模型、策略、仓储接口、领域服务
- `infrastructure`：JDBC 仓储、配置、异常处理、外部集成

从当前控制器与服务结构看，后端已经落地以下核心能力：

- Session / Workflow 主链路
- Task run / submit
- Practice 题目生成、查询、提交
- Learning Feedback 与 Weak Point Diagnosis
- Tutor Message 持久化与交互
- LLM Prompt、解析、校验、评估器相关能力

这意味着后端已经具备“主流程可跑、关键数据可落库、策略可继续迭代”的基础。

### 3.2 数据库现状

数据库迁移目前已推进到 `V17`，说明项目已经跨过最初的表结构搭建阶段，正在围绕业务模型持续补齐和收敛。

当前已持久化的关键能力包括：

- 学习会话与知识节点
- 任务与任务尝试记录
- 掌握度与证据留痕
- 用户与会话历史
- Tutor 消息
- 练习题、练习提交、测验与学习反馈
- LLM 可观测性与配置相关数据

这套数据基础已经足够支撑 MVP 级的闭环追踪、结构化沉淀和后续策略演进。

### 3.3 前端现状

前端基于 Vue 3、Pinia、Vue Router、TypeScript 与 Vite，当前已经不是静态页面集合，而是一个可以承接主业务流程的前端壳层。

目前前端已经具备：

- 登录页、首页、会话页、任务页、历史页
- `session`、`workflow`、`practice`、`tutor`、`auth` 等状态管理
- `auth`、`session`、`task`、`practice`、`learningFeedback`、`tutor` 等 API 接入
- 进度条、训练卡片、Tutor 面板、会话摘要等业务组件

说明前端已经能够承接从“进入系统”到“执行学习任务”再到“训练反馈”的主要交互流程。

### 3.4 测试与工程化现状

当前后端测试已经覆盖到一些关键能力：

- 任务执行
- 训练提交
- 练习服务与作答服务
- 掌握度计算与薄弱点诊断
- 个性化路径规划
- Prompt 模板、LLM 解析与评估逻辑
- CORS 预检集成测试

仓库中也已存在 CI/CD workflow 与部署配置，因此项目已经具备基础工程交付能力，而不是只能本地演示。

## 4. 综合判断

### 4.1 当前成熟度

如果用一句话概括当前阶段，更准确的表述是：

`MVP 主链路已成型，但产品闭环感和一致性仍在强化中`

也就是说，项目已经不是“空框架”，但也还没有完全收敛成一套表达清晰、体验稳定、叙事统一的成熟 MVP。当前阶段的特点是：

- 业务骨架已经可用
- 核心数据模型已经成型
- 工程基础已经足够支撑持续迭代
- 下一阶段的价值更偏向闭环增强、一致性收敛和体验打磨

### 4.2 当前主要风险

- 产品语义已经领先于界面表达，能力虽然存在，但用户未必能明显感知自适应学习闭环。
- 历史演进痕迹仍然留在命名、阶段定义和部分模型边界上。
- Rule / LLM / fallback 的机制已经存在，但并非所有业务决策都足够可解释。
- 跨模块一致性仍较依赖人工维护，文档、代码与接口契约的同步成本不低。

## 5. 下一步建议重点

如果下一阶段目标不是“继续堆功能”，而是“让 MVP 更像一个完整产品”，最值得优先做的是：

1. 统一后端、前端、文档中的术语和阶段模型
2. 强化“训练结果 -> 下一步学习动作”的可见闭环
3. 把分散能力收敛成一条更清晰的 MVP 用户故事
4. 继续补关键状态流转、多步骤业务链路的测试

## 6. 一句话总结

这个项目已经具备一个可信的 AI 学习流程导航系统 MVP 基础，接下来的重点不再是大面积扩功能，而是把现有能力整理成一个更清晰、更可感知、也更一致的个性化学习闭环。
