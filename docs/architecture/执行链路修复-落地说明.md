# 执行链路修复（增量落地说明）

本文档用于把“执行链路修复设计稿”转换成可直接编码与验收的落地约束清单，确保不会再出现“上游产出结构化对象、下游却不消费”的断裂。

## 1. 分层职责边界（落地版）

### 1.1 Goal
- **产出对象**：`StructuredLearningGoal`、`GoalContextSnapshot`
- **下游必须消费**：
  - `StructuredLearningGoal.goalType/subject/topics/timeBudget/selfReportedLevel/preferenceTags`
  - `GoalContextSnapshot.requiresDiagnosis/planningMode/entryGranularity/strategyHints/riskTags/explanationFocus`

### 1.2 Diagnosis
- **产出对象**：`LearnerProfileSnapshot`、`DiagnosisEvidenceSummary`、`LearnerStrategyProfile`（新增）
- **下游必须消费**：
  - `LearnerProfileSnapshot.foundationLevel/executionStability/timeBudgetLevel/learningPreference/blockerTags/riskTags/urgencyLevel`
  - `DiagnosisEvidenceSummary.primaryGapType/primaryRiskTags/keyEvidence`
  - `LearnerStrategyProfile.preferredTaskTypes/scaffoldIntensity/feedbackStyle/checkpointFrequency/riskMitigationTags`

### 1.3 Plan Preview（PlanDraft）
- **产出对象**：`LearningPlanPreview(TaskBlueprint[])` + `ExecutableTaskSpec[]`（新增）
- **下游必须消费**：
  - `ExecutableTaskSpec.completionCriteria/evidenceToCollect/evaluationRubric/scaffoldPolicy`

### 1.4 Plan Commit / Session
- **产出对象**：`LearningPlan`（commit snapshot）+ session 运行态（进度/当前任务/状态）
- **下游必须消费**：
  - `taskSequence/currentTaskIndex/status`

### 1.5 Current Task
- **产出对象**：当前 `ExecutableTaskSpec` + `TaskExecutionContext`（新增，执行期上下文）
- **下游必须消费**：
  - `TaskExecutionContext.goalContextSnapshot.explanationFocus/riskTags`
  - `TaskExecutionContext.learnerProfileSnapshot.executionStability/timeBudgetLevel`
  - `TaskExecutionContext.learnerStrategyProfile.scaffoldIntensity/feedbackStyle/checkpointFrequency`
  - `TaskExecutionContext.executableTaskSpec`（全部执行约束字段）

### 1.6 Task Scaffold
- **产出对象**：`TaskScaffold`（展示+模板）+（可选）执行态快照
- **下游必须消费**：
  - `completionSignals/antiPatterns/suggestedExploreTurns/suggestedCheckpointCount`

### 1.7 Task Messages
- **产出对象**：`TutorTurnResult`（新增，每回合结构化结果）+ `TaskExecutionRuntime`（权威运行态）
- **下游必须消费**：
  - `TaskExecutionRuntime.state` + runtime 的证据与迁移记录
  - `TutorTurnResult.semanticSummary/evidenceExtracts/detectedAction/fallbackMode`

## 2. 规则化 vs LLM 辅助（强制边界）

### 2.1 必须规则化（不可交给 LLM）
- **时间预算约束**：总时长上限、每任务 timebox、超时降级策略（压缩阶段/跳转检查/收敛总结）
- **任务数约束**：`minTasks/maxTasks` 与 `taskSequence` materialize 约束
- **阶段压缩规则**：压缩触发条件与 `compressionReason`
- **任务类型映射**：goal/profile/strategy → `TaskType` + `ExecutableTaskSpec.scaffoldPolicy`
- **状态迁移**：`TaskExecutionState` 迁移条件（回合数/检查点/证据齐备/时间预算）
- **完成判定**：基于 `ExecutableTaskSpec.evaluationRubric` 与 runtime 证据链的硬门槛

### 2.2 允许 LLM 辅助（必须可降级）
- **任务文案润色**：不改变结构化含义，只提升表达
- **教学解释生成**：围绕 `explanationFocus/taskMethod` 生成解释段落
- **用户回答语义归纳**：生成 `TutorTurnResult.semanticSummary` 与证据抽取候选
- **追问建议**：输出候选集，最终选择受 `scaffoldPolicy/checkpointFrequency` 约束

### 2.3 降级策略（必须具备）
- LLM 超时/失败：回落到 `MockLlmGateway` 或模板化回复；**不影响状态机推进与完成判定**。
- 任何 LLM 输出只能写入解释层字段（如 `assistantReply/explanationDraft/suggestedFollowups`），不得越权修改主决策字段。

## 3. 增量修复原则（不推翻现有架构）
- **先合同，后实现**：新增对象合同稳定后再贯通到各 service/DTO。
- **复用现有主对象**：`StructuredLearningGoal/GoalContextSnapshot/LearnerProfileSnapshot/LearningPlanPreview/TaskBlueprint` 保留，仅补齐缺失语义。
- **决策与展示分离**：`ExecutableTaskSpec/TaskExecutionContext/TaskExecutionRuntime` 为执行合同；`TaskBlueprint/TaskScaffoldResponse` 为展示与引导。
- **最小切口**：优先修 execution（runtime/persistence/guard/flow），再把 planning/diagnosis 的新增字段接入。
- **接口向后兼容**：现有 API 字段不删；新增字段只追加。
- **读取路径一致性**：guard 的存在性校验与实际读取来源保持一致，并逐步补齐 DB fallback（重启可恢复）。

## 4. 推荐开发顺序（建议）
1. 冻结新增对象合同：`LearnerStrategyProfile/ExecutableTaskSpec/TaskExecutionContext/TutorTurnResult`
2. 执行期上下文装配器：执行侧统一从 `TaskExecutionContext` 读
3. 强化 `TaskExecutionRuntime` 留痕：transition/evidence 结构化
4. 完成判定规则化并与 rubric 对齐
5. 规划侧产出 `ExecutableTaskSpec` 并随 commit materialize
6. `current-task` 响应补齐执行合同与上下文摘要，减少漏消费
7. diagnosis 侧规则化 `LearnerStrategyProfile` 并沉淀证据
8. guard/读取路径一致性与 DB fallback 收尾
