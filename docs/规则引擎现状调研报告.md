# 规则引擎现状调研报告

> 调研日期：2026-03-24
> 调研范围：`backend/src/main/java/navigator/application`、`backend/src/main/java/navigator/domain/policy`、相关测试与现有文档

## 1. 结论摘要

当前项目的“规则引擎”并不是一个独立的通用 Rule Engine 中间件，而是一组分布在应用层、领域策略层中的规则化决策组件。整体设计遵循“规则主导，LLM 辅助”的思路，主链路上的关键判断基本都由代码规则完成，包括：

- 目标解析与上下文派生
- 诊断答案归一化与学习者画像生成
- 学习规划策略选择与模板化任务生成
- 任务执行期的引导相位切换、交互策略约束
- 结课报告与 Next Action 决策

从工程现状看，这套规则系统已经支撑起 MVP 主链路，优点是可控、稳定、可测试；不足是规则高度硬编码、字符串常量较多、配置化与可扩展性一般，且部分规则开始混入展示型特判与流程状态耦合。

## 2. 规则引擎全景

### 2.1 规则在主链路中的位置

主链路可以概括为：

`GoalInput -> StructuredLearningGoal -> GoalContextSnapshot -> LearnerProfileSnapshot -> PlanningContext -> LearningPlanPreview -> TaskScaffold/GuidanceDecision -> LearningReport/NextActionDecision`

对应主要组件如下：

| 阶段 | 核心组件 | 作用 |
| --- | --- | --- |
| Goal | `GoalRuleEngine` | 从用户输入推导目标类型、主题范围、紧迫度、预期深度等 |
| Goal Context | `GoalContextDeriver` | 从结构化目标继续推导规划模式、入口粒度、风险标签、策略提示 |
| Diagnosis | `DiagnosisAnswerNormalizer`、`DiagnosisRuleEngine`、`LearnerStrategyProfileDeriver` | 诊断答案归一化，生成学习者画像和执行策略画像 |
| Planning | `PlanningContextAssembler`、`PlanStrategySelector`、`PlanTemplateFactory`、`TimeBudgetEnforcer` | 组装规划上下文、选策略、出任务模板并做时间约束 |
| Task Guidance | `LearningActionDetector`、`GuidancePhaseTransitionEvaluator`、`TutorInteractionPolicy`、`TaskGuidanceEngine`、`CompletionEvaluator`、`TaskScaffoldFactory` | 识别用户行为、控制引导相位、限制回答方式、判断完成情况 |
| Report | `SessionEvidenceAggregator`、`ReportApplicationService` | 汇总执行证据并产出报告与下一步动作 |

### 2.2 调用链特点

1. 规则主要以内聚 Java 类实现，而非外部 DSL/YAML/数据库规则表。
2. 应用服务承担“编排 + 调用规则 + 持久化”职责。
3. 规则结果普遍沉淀为结构化对象，供后续阶段继续消费，而不是一次性文本输出。
4. LLM 在当前后端实现中更多承担讲解/辅导补充角色，不主导枚举型决策。

## 3. 各模块规则分析

### 3.1 目标规则层

核心类：

- `backend/src/main/java/navigator/application/goal/GoalRuleEngine.java`
- `backend/src/main/java/navigator/application/goal/GoalKeywordExtractor.java`
- `backend/src/main/java/navigator/application/goal/GoalContextDeriver.java`

#### 规则职责

`GoalRuleEngine` 负责把 `LearningGoalInput` 转成 `StructuredLearningGoal`，当前规则包括：

- 根据关键词推导 `GoalType`
  - 如考试/复习/冲刺 -> `REVIEW_FOR_EXAM`
  - “不会/卡住” -> `FIX_SPECIFIC_BLOCKER`
  - “做题/练习” -> `PRACTICE_ENHANCEMENT`
  - “系统/全面/408” -> `BUILD_SYSTEMATIC_UNDERSTANDING`
  - 否则回落为 `LEARN_NEW_CONCEPT`
- 根据文本和 hints 提取 topics
- 推导 `subject`
  - `408` -> `COMPUTER_408`
  - 数据结构主题 -> `DATA_STRUCTURE`
  - 否则 `GENERAL_CS`
- 推导 `topicScopeType`
  - chapter/course/single/multi
- 根据时间预算和关键词推导 `UrgencyLevel`
- 根据目标类型和时间预算推导 `expectedDepth`

`GoalContextDeriver` 进一步派生：

- `PlanningMode`
  - 高紧迫考试 -> `EXAM_CRASH`
  - 系统性目标/章节课程范围 -> `SYSTEMATIC_BUILD`
  - 练习驱动或偏好练习优先 -> `PRACTICE_DRIVEN`
  - 初学者 -> `STEADY_FOUNDATION`
  - 默认 -> `CONCEPT_CLARIFICATION`
- `EntryGranularity`
  - 短时预算 -> `MICRO`
  - 初学者、多主题 -> `SMALL`
  - 课程级且长期 -> `MEDIUM`
- 风险标签
  - `TIME_PRESSURE`
  - `GOAL_TOO_BROAD`
  - `GOAL_TOO_VAGUE`
  - `OVERCONFIDENCE_RISK`
- 策略提示和解释焦点

#### 当前评价

这一层规则比较清晰，已经形成“输入归一化 -> 上下文补全”的两级结构，是当前规则体系里最完整的一段。

#### 主要限制

- 关键词和 topic 字典全部硬编码在类里，扩展新学科成本高。
- `subject`、`topicScopeType`、风险标签大量使用字符串，而非强类型枚举。
- 文本规则偏中文关键词匹配，鲁棒性依赖表达方式。

### 3.2 诊断规则层

核心类：

- `backend/src/main/java/navigator/application/diagnosis/DiagnosisAnswerNormalizer.java`
- `backend/src/main/java/navigator/application/diagnosis/DiagnosisRuleEngine.java`
- `backend/src/main/java/navigator/application/diagnosis/LearnerStrategyProfileDeriver.java`

#### 规则职责

`DiagnosisAnswerNormalizer` 做两件事：

- 将 questionId alias 归一为标准 questionId
- 从多题答案里提取首选 code，形成 `NormalizedAnswers`

`DiagnosisRuleEngine` 基于标准答案、目标、目标上下文生成 `LearnerProfileSnapshot`：

- 基础水平映射：`BEGINNER/BASIC/PROFICIENT/... -> FoundationLevel`
- 主阻塞点映射：`CONCEPT_UNCLEAR` 等别名归一到统一 gap code
- 执行稳定度推导：
  - 初学者、低信心+关键概念缺口 -> `UNSTABLE`
  - 时间压力/认知负荷 -> `MODERATE`
  - 其余 -> `STABLE`
- 学习偏好推导：
  - 优先用诊断答案
  - 没有则回退到目标偏好标签
- 风险标签推导：
  - `PREREQUISITE_GAP`
  - `TIME_PRESSURE`
  - `OVERCONFIDENCE_RISK`
  - `SHALLOW_UNDERSTANDING_RISK`
  - 以及 execution risk code

`LearnerStrategyProfileDeriver` 再把画像转成执行策略画像：

- `ScaffoldIntensity`
- `FeedbackStyle`
- `checkpointFrequency`
- `preferredTaskTypes`
- `riskMitigationTags`

#### 当前评价

诊断层已经把“诊断结果”与“执行策略”分开，层次比单纯打标签更好，这一点对后续规划与任务执行复用很有价值。

#### 主要限制

- `NormalizedAnswers` 中的 `goalOutcomeCode`、`weaknessScopeCode` 目前没有被充分消费。
- 当前归一化逻辑基本只取每题首个选项，暂不支持多选组合推理。
- 风险与 gap code 仍然大量依赖字符串常量。

### 3.3 规划规则层

核心类：

- `backend/src/main/java/navigator/application/planning/PlanningContextAssembler.java`
- `backend/src/main/java/navigator/application/planning/PlanStrategySelector.java`
- `backend/src/main/java/navigator/application/planning/PlanTemplateFactory.java`
- `backend/src/main/java/navigator/application/planning/TimeBudgetEnforcer.java`

#### 规则职责

`PlanningContextAssembler` 负责从 store 中把各阶段产物组装为 `PlanningContext`，同时派生 `TimeBudgetConstraint`：

- 不同 `TimeBudget` 映射为不同 `totalMinutesCap`
- 根据预算和粒度推导 `maxTasks`
- 对少数 showcase 场景做特判，强制提高最少任务数量

`PlanStrategySelector` 是规划层最核心的策略矩阵：

- 章节/课程范围、系统化目标 -> `FRAMEWORK_BUILD`
- 初学者或有前置缺口 -> `FOUNDATION_PATCH`
- 考前高紧迫复习 -> `SPRINT_CORRECTION`
- 程序/题型缺口或练习提升目标 -> `DRILL_STRENGTHEN`
- 单点卡点修复 -> `LOCAL_REPAIR`
- 默认 -> `CONCEPT_CLARIFICATION`

`PlanTemplateFactory` 根据策略生成阶段和任务模板：

- 每种策略对应固定的 stages + `TaskBlueprint[]`
- 会根据 `LearnerStrategyProfile.preferredTaskTypes` 调整任务排序
- 会调用 `TimeBudgetEnforcer` 对任务数量和时长做裁剪

`TimeBudgetEnforcer` 的规则相对简单：

- 若总时长和任务数超预算，先裁任务数
- 再按平均时长重算每个任务分钟数

#### 当前评价

规划规则已经具备“策略选择 -> 模板产出 -> 时间压缩”的完整闭环，工程上很适合 MVP。

#### 主要限制

- `PlanningContextAssembler` 中存在明显的 showcase 特判逻辑，说明规划规则层已混入前端展示需求。
- `PlanTemplateFactory` 仍是模板工厂，不是可组合规则；新增策略需要改代码。
- `TimeBudgetEnforcer` 的压缩策略较粗，只做平均分配，不区分任务优先级。

### 3.4 任务执行与引导规则层

核心类：

- `backend/src/main/java/navigator/application/learning/LearningActionDetector.java`
- `backend/src/main/java/navigator/application/task/guidance/GuidancePhaseTransitionEvaluator.java`
- `backend/src/main/java/navigator/domain/policy/tutor/TutorInteractionPolicy.java`
- `backend/src/main/java/navigator/application/task/guidance/TaskGuidanceEngine.java`
- `backend/src/main/java/navigator/application/task/TaskScaffoldFactory.java`
- `backend/src/main/java/navigator/application/task/CompletionEvaluator.java`

#### 规则职责

这一层本质上是“轻量状态机 + 策略表”。

`LearningActionDetector`

- 基于关键词把用户输入识别为：
  - 要例子
  - 要对比
  - 要简化
  - 困惑信号
  - 直接要答案
  - 自我解释
  - 泛化输入

`GuidancePhaseTransitionEvaluator`

- 仅根据 `exploreTurnCount` 线性切换：
  - 1 -> `CLARIFY_GOAL`
  - 2 -> `BUILD_FRAME`
  - 3 -> `TRY_EXPRESS`
  - 4 -> `PROBE_GAPS`
  - 5 -> `META_REFLECT`
  - 6+ -> `TRANSITION_HINT`

`TutorInteractionPolicy`

- 根据 phase + user action + strategy，输出 `GuidanceDecision`
- 内置了几条最关键的策略约束：
  - 阻止直接给答案
  - 跑题时拉回任务
  - 用户困惑时追问定位
  - 用户主动提问时给 scaffold hint 而非代做
- 同时也定义了 phase 默认行为

`TaskScaffoldFactory`

- 根据 `TaskBlueprint` 构造执行期脚手架
- 固化 4 个认知单元：
  - understand
  - explore
  - self_explain
  - verify
- 根据 `ScaffoldIntensity` 调整建议探索轮次和 checkpoint 次数

`CompletionEvaluator`

- 若任务自带 rubric，则按维度校验
- 否则回退到最小字符数与关键词命中规则

#### 当前评价

任务层是当前项目里最像“策略引擎”的部分，尤其 `TutorInteractionPolicy` 已经接近规则表模型。相比前几层的静态分类，这一层已经开始处理执行中的动态行为。

#### 主要限制

- 引导相位只看轮次，不看真实理解质量，粒度偏粗。
- `TutorInteractionPolicy` 注释中已经写明“后续可换 YAML 加载”，说明当前实现只是第一版硬编码策略表。
- `CompletionEvaluator` fallback 规则较弱，容易被“凑字数”绕过。
- 行为识别依然是关键词级别，误判和漏判空间较大。

### 3.5 报告与 Next Action 规则层

核心类：

- `backend/src/main/java/navigator/application/report/SessionEvidenceAggregator.java`
- `backend/src/main/java/navigator/application/ReportApplicationService.java`

#### 规则职责

这一层负责把执行证据再结构化一次，并生成结束态判断：

- 汇总完成任务数、时长、交互次数、总结提交情况、问题标签
- 计算 closure quality、直接答案依赖度等指标
- 根据完成度、缺口、学习方法画像，输出：
  - `ResultStatus`
  - `NextActionType`
  - `requiresReplan`
  - 文本化 reason / nextEntryPoint

#### 当前评价

这部分不是传统意义上的“规则引擎入口”，但已经承担了明显的规则决策职责，可视为规则体系的收束层。

#### 主要限制

- 与前面一样，仍然大量依赖 hard-coded if/else。
- `NextAction` 判定逻辑已较长，后续若继续增长，建议独立拆分为专门决策器。

- `PlanningContextAssemblerTest`

整体上，目标、诊断、规划三段的核心规则已经有单元测试兜底；任务引导层也覆盖了若干关键组件。

相对薄弱的地方：

- `TutorInteractionPolicy` 缺少直接单测
- `GuidancePhaseTransitionEvaluator` 缺少直接单测
- Report / Next Action 的判定没有看到专门规则级测试
- 规则之间的跨阶段组合测试不算多

## 5. 当前规则引擎的工程特点

### 5.1 优点

- 主链路完整，已经能独立支撑 MVP。
- 决策可解释，排查问题时能直接定位到代码分支。
- 输出对象结构化，便于阶段间复用。
- 与 LLM 解耦较好，LLM 不在线时核心流程仍可运行。
- 测试基础比很多早期项目要扎实。

### 5.2 主要问题

#### 1. 规则强依赖硬编码

规则散落在多个 Java 类中，新增一条规则通常需要改代码、改测试、重新发版，不利于运营侧或产品侧快速调整。

#### 2. 字符串常量过多

如 `TIME_PRESSURE`、`GOAL_TOO_BROAD`、`SINGLE_TOPIC`、`CONCEPT_GAP` 等大量依赖字符串拼写，一旦多处复制，维护成本和出错概率都会提高。

#### 3. 泛化规则与 showcase 特判混杂

`PlanningContextAssembler` 里对“数组 vs 链表”“二叉树”“DFS vs BFS”做了单独特判。这类逻辑更像产品演示兼容，不应长期停留在通用规划规则核心层。

#### 4. 状态来源混合

规划与执行规则大量依赖 `InMemoryStore` 拼装上下文，同时又写数据库，说明当前系统处于“内存态 + 持久化态并存”的过渡阶段。对规则重放、排查和横向扩展都不是最理想状态。

#### 5. 部分字段进入链路但未充分使用

例如诊断归一化里的 `goalOutcomeCode`、`weaknessScopeCode` 目前价值没有完全释放，说明数据模型先走在了规则消费前面。

#### 6. 动态能力仍偏弱

任务引导虽然已经进入执行态，但核心切换仍主要基于轮次、字符数、关键词，不是真正的证据驱动。

## 6. 演进建议

### 6.1 短期建议

1. 把高频字符串标签逐步收敛为枚举或常量类。
2. 给 `TutorInteractionPolicy`、`GuidancePhaseTransitionEvaluator`、`ReportApplicationService` 补齐规则级测试。
3. 把 showcase 特判从通用规则层抽离，收口到单独的 demo compatibility 层。
4. 在调研文档之外，补一张“规则输入/输出对象图”，方便后续接手。

### 6.2 中期建议

1. 将 `TutorInteractionPolicy` 先配置化，优先尝试 YAML/JSON 策略表。
2. 对规划策略矩阵做“规则表化”，把条件和策略映射从 if/else 中抽出。
3. 让 `CompletionEvaluator` 更多消费结构化证据，而不是只看字数。
4. 让任务引导相位从“按轮次切换”逐步升级为“按证据切换”。

### 6.3 长期建议

1. 建立统一规则元模型：
   - 输入上下文
   - 命中条件
   - 输出决策
   - 优先级
   - 解释文本
2. 将 Goal/Diagnosis/Planning/Guidance/Report 五段规则纳入统一治理方式。
3. 为规则变更建立回放集和回归基线，避免经验性调整破坏已有链路。

## 7. 最终判断

如果从“是否已经有规则引擎”这个问题来看，答案是：有，而且已经覆盖了项目主链路的核心决策。

如果从“是否已经形成可配置、可运营、可扩展的统一规则平台”来看，答案是：还没有，目前更准确的定义应当是“以多个规则化决策组件组成的应用层规则系统”。

这套系统当前最适合的定位不是推倒重做，而是：

- 保留现有规则主导架构
- 先补齐测试和边界梳理
- 再逐步把硬编码规则抽象为可配置规则表

这样风险最小，也最符合项目当前阶段。
