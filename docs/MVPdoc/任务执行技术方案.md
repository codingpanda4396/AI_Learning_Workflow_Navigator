# 任务执行技术方案

## 1. 文档目标

本文档用于明确 AI 学习导航系统中“任务执行（Task Execution）”模块的职责边界、设计原则、任务单元结构、内嵌 LLM 导师协作方式、执行证据采集机制，以及与《目标输入技术文档》《用户诊断技术方案》《学习规划技术方案》的衔接关系。

该文档与前三个模块协同工作：

- **目标输入模块**负责把用户当前学习意图转化为结构化目标快照，回答“用户现在想学什么、在什么约束下学”
- **用户诊断模块**负责识别当前目标下的用户状态、阻塞点与执行风险，回答“用户现在以什么状态去学、最可能卡在哪里”
- **学习规划模块**负责把“目标 + 状态 + 约束”编译为一份可执行的推进方案，回答“系统接下来应该如何推进”
- **任务执行模块**负责把规划结果落地为一组可推进、可对话、可检查、可留痕的学习任务，回答“用户这一轮具体要怎么学、怎么与 LLM 协作、系统如何判断是否真正推进了学习”

因此，任务执行模块不是知识内容的直接灌输层，也不是一个自由聊天框，而是系统闭环中的：

> **学习行为编排层 + 对话脚手架层 + 执行证据采集层**

---

## 2. 模块定位

在系统最小闭环中，任务执行位于第四环：

**目标输入 → 用户诊断 → 学习规划 → 任务执行 → 反馈与 Next Action**

任务执行模块承担的核心职责不是“把知识讲完”，而是：

1. **把学习规划模块产出的策略与路径落实为一组具体任务**
2. **为每个任务提供框架化的学习目标、提问方式与完成标准**
3. **让用户在受约束的 LLM 导师辅助下完成具体知识学习与理解建构**
4. **将执行过程沉淀为后续反馈与动态调整可消费的证据**

换句话说：

- 学习规划回答“应该怎么安排路径”
- 任务执行回答“当前这一步具体怎么推进”

本模块的产品价值不在于替用户输出大量知识细节，而在于：

> **教用户如何高质量地使用 LLM 来学习，而不是替用户把学习过程做完。**

这与项目整体定位保持一致：

> **我们解决的不是“用户缺一个会回答问题的大模型”，而是“用户不会把 LLM 用成真正有效的学习工具”。**

---

## 3. 设计边界

### 3.1 不是知识内容生成器

本模块不追求自动生成整章知识讲义，也不追求替代教材、课程视频或题解平台。

系统不应把执行页设计为：
- 大段概念讲解页
- 纯内容阅读页
- 自动生成整套答案页

否则会退化为普通 AI 问答或内容生成产品，无法体现“学习导航”的价值。

### 3.2 不是自由聊天机器人

本模块中的内嵌 LLM 导师不是一个无限开放的聊天窗口。

它必须带着当前任务的上下文运行，包括：
- 当前任务目标
- 当前任务类型
- 当前用户画像摘要
- 当前推荐提问框架
- 当前任务边界与禁止行为

这样才能保证“导师对话”服务于学习任务，而不是偏离学习主线。

### 3.3 不是只靠用户主观点击完成的流程页

任务执行不能仅靠“点完成”来判断学习是否推进。

系统必须尽量采集：
- 用户是否真的进行了对话或输出
- 用户是否完成了任务要求的动作
- 用户是否暴露出新的薄弱点
- 本轮是否达到可接受的完成质量

### 3.4 是“学习方法产品化”的落地点

任务执行模块真正产品化的，不是知识本身，而是：

- 如何围绕一个知识点建立学习框架
- 如何向 LLM 提出高质量问题
- 如何进行追问、复述、比较与纠偏
- 如何判断自己是否真的理解

因此，本模块是“教道而非术”的最直接承载层。

## 4. 与前三个模块的协同关系

任务执行模块不能独立设计，它必须严格消费前三个模块的结果。

### 4.1 对上承接目标输入模块

目标输入模块已经提供了：
- 当前学习目标
- 学习范围与知识锚点
- 时间预算
- 学习偏好
- 紧迫程度
- 预期掌握深度

任务执行模块不需要重新询问“你想学什么”，而应基于这些信息控制：
- 当前任务时长
- 任务数量
- 任务展开深度
- 对话节奏

### 4.2 对上承接用户诊断模块

用户诊断模块已经提供了：
- 当前基础层级
- 核心阻塞点
- 风险标签
- 粒度偏好
- 反馈频率建议
- 规划提示信号

任务执行模块应将这些信号进一步落实为：
- 当前任务应该偏概念、偏对比还是偏练习
- 当前任务是否应提供更多提示与脚手架
- 是否应插入自解释、校准、纠偏类动作

### 4.3 对上承接学习规划模块

学习规划模块已经产出了：
- 推荐入口点 `recommendedEntry`
- 规划主策略 `strategyCode`
- 任务粒度 `taskGranularity`
- 任务类型配比 `taskMix`
- 检查点密度 `checkpointDensity`
- 反馈频率 `feedbackFrequency`
- 保底路径 / 回退路径 `fallbackPath`

任务执行模块不应重新做宏观决策，而应把这些结果转译为：
- 当前执行会话
- 当前任务序列
- 当前任务单元配置
- 当前任务的完成条件
- 当前阶段的切换条件

### 4.4 对下服务反馈与 Next Action 模块

任务执行模块输出给下一模块的，不应只是“任务完成/未完成”，而应包括：
- 用户实际进行了哪些学习动作
- 用户在哪些点上反复追问或停滞
- 用户输出是否达到任务目标
- 新暴露出的概念缺口、应用缺口或表达缺口
- 是否应继续、加练、回退或切换路径

因此，任务执行是整个闭环中最关键的“证据生成层”。

## 5. 核心设计原则

### 5.1 任务是“学习动作”，不是“知识块”

任务执行页中的最小单元不应是“知识内容片段”，而应是一个清晰、可完成、可检查的学习动作。

例如不应只写：
- 学习链表

而应改写为：
- 用自己的话解释链表节点为什么需要 `next` 指针
- 让导师用一个最小例子演示单链表插入时指针如何变化
- 比较数组和链表在存储方式上的 3 个核心差异
- 完成 1 道小题并说明为什么这里更适合用队列而不是栈

系统要引导用户“做动作”，而不是“看一眼内容”。

### 5.2 系统提供的是学习框架与提问方式

任务执行模块的核心不是提供知识正文，而是提供：
- 当前任务目标
- 推荐提问方式
- 推荐追问路径
- 完成标准
- 自检方式

也就是说，系统要把“如何学、如何问、如何检验”产品化。

### 5.3 内嵌导师必须受任务约束

LLM 导师不能无限展开，也不能一上来直接输出完整答案。

导师行为必须受以下约束：
- 围绕当前任务目标回答
- 优先引导、比较、追问、纠偏
- 在需要时提供最小例子或步骤拆解
- 尽量避免脱离当前任务边界的大段扩展

### 5.4 执行过程必须可观测

系统必须尽量把执行过程转化为可记录的行为信号，例如：
- 用户是否发起了有效提问
- 用户是否完成了自我解释
- 用户是否完成了微练习
- 用户在哪些点停留时间较长
- 用户在哪些概念上表现出混淆

### 5.5 执行结果必须可解释、可回溯、可调整

后续反馈与动态调整必须能追溯：
- 这个任务为什么被判定为已完成/未完成
- 用户在哪一步被判断为“已理解”或“仍存在缺口”
- 下一步为什么建议继续、回退或加练

---

## 6. 任务执行模块要解决的核心问题

本模块本质上回答以下四个问题：

### 6.1 用户当前这一步具体要做什么？

不是泛泛地“继续学习”，而是明确：
- 当前任务是什么
- 目标是什么
- 预期花多久
- 当前任务与整体路径的关系是什么

### 6.2 用户应该如何借助 LLM 学这一小步？

系统要给出：
- 推荐的提问方式
- 推荐的追问路径
- 适合当前阶段的交互范式
- 什么时候应让导师讲解，什么时候应让用户先说

### 6.3 系统如何判断用户是否真的推进了学习？

不能只看是否点击了完成按钮，而要看：
- 是否有实际互动
- 是否有可验证输出
- 是否满足最低完成标准
- 是否仍然暴露出关键性错误

### 6.4 执行过程中产生了哪些对下一步有价值的证据？

例如：
- 用户反复卡在某个前置概念
- 用户会描述概念但不会应用
- 用户会做题但解释不清
- 用户主观感觉很懂，但微测暴露出误解

这些证据是反馈与动态调整的直接输入。

---

## 7. 任务执行的总体分层设计

为了保证该模块既能落地、又能解释清楚，建议将任务执行拆成四层：

### 7.1 第一层：执行会话层

负责管理“这一轮学习执行现在推进到哪里”，包括：
- 当前执行会话标识
- 当前阶段
- 当前任务
- 已完成任务数
- 当前状态（进行中 / 已暂停 / 待反馈 / 已完成）

这一层解决“任务推进上下文”的问题。

### 7.2 第二层：任务编排层

负责将学习规划模块产出的路径结果，转译成一组按顺序执行的任务单元。

这一层主要负责：
- 生成任务序列
- 控制任务类型组合
- 设置每个任务的时长、粒度与检查点
- 定义阶段切换条件

### 7.3 第三层：对话脚手架层

负责为每个任务生成：
- 任务目标说明
- 提问模板
- 推荐追问方式
- 完成标准
- 自检提醒
- 导师行为约束

这一层是你所强调的“教用户如何用 LLM”的关键落点。

### 7.4 第四层：证据采集层

负责沉淀：
- 对话摘要
- 用户输出
- 微练习结果
- 导师评估
- 难点标签
- 下一步建议信号

这一层是后续反馈与 Next Action 的数据基础。

---

## 8. 为什么任务执行不能直接讲知识细节

这是本模块设计中必须明确的一点。

### 8.1 否则会退化为通用 AI 问答工具

如果系统只是不断输出“链表是什么”“队列是什么”“DFS 怎么做”，那么用户看到的只是一个换皮问答系统，而不是学习导航系统。

### 8.2 不利于形成方法层差异化

你们真正有价值的地方不是“知道知识”，而是：
- 知道这一步该先问什么
- 知道怎样追问才能更快理解
- 知道怎样验证自己有没有懂
- 知道下一步是继续还是回退

这些都属于“学习方法产品化”，而不是“知识搬运”。

### 8.3 不利于形成闭环证据

如果执行页只是看内容，很难判断：
- 用户到底有没有理解
- 用户在哪一步产生了阻塞
- 用户提问质量如何
- 应该如何调整下一步

而“任务化 + 对话脚手架 + 输出要求”能更自然地产生结构化证据。

## 9. 核心执行对象设计

为了与前三个模块保持同样的工程表达风格，任务执行模块建议至少包含以下核心对象。

---

### 9.1 ExecutionSession

表示一次学习规划落地后的执行会话，用于跟踪当前学习推进状态。

#### Java 示例

```java
public class ExecutionSession {
    private String executionSessionId;
    private String planId;
    private String sessionId;
    private String goalId;

    private String currentStageCode;
    private String currentTaskId;
    private ExecutionStatus status;

    private Integer completedTaskCount;
    private Integer totalTaskCount;

    private LocalDateTime startedAt;
    private LocalDateTime lastActiveAt;
}
```

#### 作用说明

该对象主要回答：
- 当前处于哪一轮执行
- 现在推进到哪个任务
- 整体完成度如何
- 当前是否需要恢复、暂停或切换阶段

---

### 9.2 LearningTask

表示一个可执行的最小任务单元，是任务执行模块的核心对象。

#### Java 示例

```java
public class LearningTask {
    private String taskId;
    private String executionSessionId;

    private String stageCode;
    private String taskType;
    private String title;
    private String objective;

    private List<String> keyConcepts;
    private List<String> scaffoldPrompts;
    private List<String> expectedOutputs;
    private List<String> completionCriteria;
    private List<String> riskHints;

    private Integer estimatedMinutes;
    private Integer orderIndex;
    private TaskStatus status;
}
```

#### 作用说明

该对象不是一段知识说明，而是完整描述：
- 当前要完成什么学习动作
- 推荐如何问导师
- 用户应该产出什么
- 什么情况下算完成
- 当前有哪些易错点或风险

---

### 9.3 TutorInteractionContext

表示传给内嵌 LLM 导师的受控上下文。

#### Java 示例

```java
public class TutorInteractionContext {
    private String taskId;
    private String goalText;
    private String stageCode;
    private String taskType;

    private String learnerLevel;
    private List<String> blockerTags;
    private List<String> riskTags;

    private String taskObjective;
    private List<String> allowedFocusPoints;
    private List<String> disallowedBehaviors;
    private List<String> suggestedPromptFrames;
}
```

#### 作用说明

该对象用于约束导师在某个任务中的行为边界，例如：
- 当前任务只围绕“链表节点与连接关系”展开
- 优先使用最小例子，不直接整段灌输
- 鼓励用户先复述，再纠偏
- 不要跳出当前任务范围讲整章内容

---

### 9.4 TaskExecutionEvidence

表示任务执行过程中产生的结构化证据。

#### Java 示例

```java
public class TaskExecutionEvidence {
    private String evidenceId;
    private String taskId;
    private String executionSessionId;

    private String evidenceType;
    private String content;

    private Boolean taskCompleted;
    private String completionQuality;
    private List<String> detectedGapTags;
    private List<String> nextActionHints;

    private LocalDateTime createdAt;
}
```

#### 作用说明

该对象是后续反馈与动态调整的重要输入，用于沉淀：
- 用户完成了什么
- 用户哪里仍然没懂
- 当前完成质量如何
- 下一步更适合继续、回退还是加练

## 10. 标准任务单元结构设计

为了让任务执行页真正承担“方法脚手架”作用，建议每个任务单元至少包含以下 6 个部分。

### 10.1 任务标题

用于告诉用户当前这一步在做什么。

示例：
- 理解链表节点与连接关系
- 比较栈和队列的使用场景
- 用一个最小例子看懂 BFS 为什么适合层序搜索

### 10.2 任务目标

用一句话明确这一步要解决的学习问题。

示例：
- 搞清楚链表为什么不依赖连续内存
- 区分“会定义”与“会应用”的差别
- 建立图搜索算法与典型题型之间的联系

### 10.3 建议提问方式

这是执行层最核心的设计内容之一。

系统不应只是让用户“去问”，而应明确告诉用户：
- 先让导师做什么
- 再让导师帮你比较什么
- 最后如何让导师检查你的理解

例如：
- 先请导师用最小例子解释核心概念
- 再请导师比较最容易混淆的两个点
- 最后用“我这样理解对吗”做一次复述校验

### 10.4 推荐追问脚手架

建议每个任务提供 3~5 个可直接点击或复制的提问模板，例如：
- 请用一个最简单的例子解释这个概念
- 这个概念最容易和什么混淆？请帮我对比
- 请不要直接给答案，先给我一步提示
- 我这样理解对不对：……
- 请结合一道典型题说明它怎么用

### 10.5 完成标准

明确告诉用户：
- 做到什么程度才算完成
- 系统会重点看哪些行为或输出

示例：
- 能用自己的话解释核心概念
- 能说出与相近概念的 2~3 条区别
- 能在一个简单例子中指出该概念如何使用

### 10.6 任务后自检

引导用户进行最低限度的元认知检查，例如：
- 如果脱离导师，你还能复述清楚吗？
- 如果换一道同类题，你知道应优先考虑什么吗？
- 你现在最模糊的地方还剩什么？

---

## 11. 任务类型体系设计

MVP 阶段不宜设计过多任务类型，但需要足够表达不同执行方式。建议优先保留以下 6 类任务类型。

### 11.1 CONCEPT_FRAME

用于建立概念框架与骨架。

#### 适用场景
- 新知识入门
- 基础薄弱
- 只知道名词但缺乏结构理解

#### 典型动作
- 给出最小定义
- 解释核心组成
- 说明为什么存在
- 说明与上下位概念的关系

---

### 11.2 COMPARE_AND_DISTINGUISH

用于处理相近概念之间的混淆。

#### 适用场景
- 概念容易混
- 学过但结构不稳
- 做题时经常用错方法

#### 典型动作
- 比较
- 区分
- 找边界
- 给反例

---

### 11.3 EXAMPLE_WALKTHROUGH

用于通过最小例子和过程拆解帮助理解机制。

#### 适用场景
- 抽象概念难懂
- 机制类知识需要动态演示
- 用户偏好 example first

#### 典型动作
- 给最小例子
- 分步骤演示
- 解释每一步发生了什么
- 指出关键状态变化

---

### 11.4 SELF_EXPLANATION

用于让用户自己输出理解，以防止“看懂幻觉”。

#### 适用场景
- 用户自信但不确定是否真懂
- 前面已经看过讲解
- 需要验证是否能独立表达

#### 典型动作
- 自己复述
- 自己解释
- 导师纠偏
- 导师指出表达中的漏洞

---

### 11.5 MICRO_PRACTICE

用于建立“概念 → 应用”的最小闭环。

#### 适用场景
- 会概念不会用
- 需要建立题型映射
- 想用最小成本检测理解程度

#### 典型动作
- 完成 1~2 道微题
- 说明为什么这么做
- 导师指出错误原因
- 对典型错法进行纠偏

---

### 11.6 REFLECTION_AND_FIX

用于收尾反思、确认问题残留并决定下一步。

#### 适用场景
- 一组任务结束后
- 发现理解仍有断层
- 需要决定继续、回退或加练

#### 典型动作
- 说出还没搞懂的地方
- 归纳当前错因
- 确认下一步建议
- 形成最小复盘记录

---

## 12. 规划模块到任务执行模块的映射方式

任务执行模块不是重新规划，而是把规划模块产出的决策结果翻译成具体任务组合。

### 12.1 典型映射关系一：基础重建路径

若学习规划模块输出的是“基础重建型策略”，则任务执行模块可映射为：

1. `CONCEPT_FRAME`
2. `COMPARE_AND_DISTINGUISH`
3. `EXAMPLE_WALKTHROUGH`
4. `SELF_EXPLANATION`
5. `REFLECTION_AND_FIX`

这种路径的特点是：
- 先稳住概念骨架
- 再清理混淆点
- 再通过例子建立动态理解
- 最后通过复述确认是否真正掌握

### 12.2 典型映射关系二：练习驱动路径

若学习规划模块输出的是“应用强化型策略”，则任务执行模块可映射为：

1. `EXAMPLE_WALKTHROUGH`
2. `MICRO_PRACTICE`
3. `SELF_EXPLANATION`
4. `REFLECTION_AND_FIX`
5. 必要时插入 `COMPARE_AND_DISTINGUISH`

这种路径更适合：
- 会概念但不会做题
- 希望尽快形成题型映射
- 需要通过少量练习快速暴露问题

### 12.3 典型映射关系三：考前压缩路径

若学习规划模块输出的是“冲刺压缩型策略”，则任务执行模块可映射为：

1. 高频框架梳理
2. 易错点对比任务
3. 最小题型校准任务
4. 错点回收任务
5. 保底清单确认任务

这种路径特点是：
- 不追求覆盖所有细节
- 优先保证高频与高收益内容
- 强调快速校准与最低风险通过

因此，任务执行模块本质上是：

> **把规划策略翻译成一组任务类型组合、任务顺序与任务约束。**

## 13. 内嵌 LLM 导师设计

这是任务执行模块最关键的能力之一。

### 13.1 导师不是独立产品，而是任务执行器的一部分

内嵌导师不应被设计为“另一个聊天页面”，而应是绑定在当前任务中的“上下文化学习助手”。

用户进入某个任务时，导师应自动获得：
- 当前任务标题
- 当前任务目标
- 当前任务类型
- 用户当前画像摘要
- 当前建议提问模板
- 当前不应越界的边界

这会显著提高导师输出的稳定性与任务相关性。

### 13.2 导师的核心职责

#### 职责 1：围绕任务目标讲解，不乱扩散

如果当前任务是“比较栈和队列”，导师就应围绕：
- 结构特点
- 进出规则
- 场景差异
- 典型混淆点

而不是突然扩展到树、图、哈希表等其他内容。

#### 职责 2：优先引导，不优先代答

尤其在以下任务类型中：
- `SELF_EXPLANATION`
- `MICRO_PRACTICE`
- `REFLECTION_AND_FIX`

导师更应先给提示、纠偏、追问，而不是直接给出标准答案。

#### 职责 3：优先使用最小例子和结构化解释

为了减少认知负担，导师应尽量使用：
- 最小例子
- 分步骤说明
- 对比式解释
- 一次只解决一个问题

#### 职责 4：帮助系统采集高价值证据

导师在交互中可以辅助判断：
- 用户是否真的理解
- 用户在哪些概念上仍存在混淆
- 当前任务完成质量如何
- 下一步更适合什么动作

这些结果可以结构化写入执行证据。

### 13.3 导师的禁止行为

为保证任务执行层不失控，建议对导师明确限制：
- 不直接把整章知识全部讲完
- 不一上来就给完整标准答案
- 不脱离当前任务边界自由扩展
- 不替用户完成应由用户自己完成的自解释动作
- 不绕开任务脚手架进入泛化问答模式

---

## 14. 任务完成判据设计

任务完成不能只依赖用户手动点击，需要有结构化判据。

### 14.1 行为完成

表示用户是否真的执行了任务要求的动作，例如：
- 与导师完成至少 2 轮有效交互
- 提交了一段自己的解释
- 完成了一道微练习

### 14.2 产出完成

表示用户是否产生了任务所要求的输出，例如：
- 给出了 3 条区别
- 解释了一个最小例子
- 说清楚了某一步为什么这样做

### 14.3 质量完成

表示当前输出是否达到最低可接受标准，例如：
- 导师判断“基本解释正确”
- 微练习达到最低正确率阈值
- 未出现关键概念性错误

MVP 阶段未必需要对“质量完成”做全自动强判断，但结构上必须预留该层，以便后续增强。

---

## 15. 执行证据采集设计

为了支撑反馈与 Next Action，任务执行模块至少应采集以下 6 类信号。

### 15.1 任务是否完成

回答：
- 用户是否推进了这一步
- 是否需要提醒、重做或跳转

### 15.2 实际耗时

回答：
- 当前任务粒度是否合理
- 用户是否在某类任务上明显耗时异常

### 15.3 用户提问集中点

回答：
- 用户真实困惑更偏概念、例子、应用还是表达
- 哪些点在不断复现

### 15.4 用户自解释质量

回答：
- 用户是否只是“看懂”而没有“能说出来”
- 是否存在高自信低表达清晰度的问题

### 15.5 微练习表现

回答：
- 用户能否从理解迁移到应用
- 是否存在“会说不会做”或“会做不会说”的断层

### 15.6 用户主观困难感

回答：
- 用户是否觉得节奏过快或过慢
- 下一步是否应压缩、细化或切换模式

---

## 16. 前端任务执行页建议结构

为了突出“学习方法脚手架”而不是“知识灌输”，建议任务执行页优先展示以下结构：

### 16.1 顶部：当前任务概览

展示：
- 当前任务标题
- 所属阶段
- 预计用时
- 任务目标
- 与整体路径的关系

### 16.2 中部左侧：任务脚手架

展示：
- 建议提问方式
- 推荐追问模板
- 当前易错提醒
- 完成标准
- 自检问题

这一块应成为页面主角，因为它体现的是“如何学习”。

### 16.3 中部右侧：内嵌导师区

导师区不是纯聊天框，而是“带任务上下文的导师面板”，支持：
- 一键使用推荐提问模板
- 保留任务上下文对话
- 支持用户提交自己的解释并请求纠偏

### 16.4 底部：任务完成与证据确认区

展示：
- 我已完成的动作
- 当前仍不懂的地方
- 是否进入下一步
- 是否需要再来一道微练习

这种结构比“大段知识讲解 + 一个聊天框”更能体现产品差异化。

---

## 17. 后端最小落地方案建议

在 MVP 阶段，任务执行模块不必一开始就做到非常复杂，建议先打通以下最小闭环。

### 17.1 最小执行链路

1. 学习规划模块产出任务序列骨架
2. 任务执行模块按骨架生成当前任务详情
3. 前端展示任务目标、提问模板、完成标准
4. 用户与内嵌导师完成当前任务交互
5. 系统采集最小证据：对话摘要 + 用户输出 + 完成状态
6. 反馈模块基于这些证据决定下一步

### 17.2 MVP 优先实现的能力

优先级建议如下：

#### 第一优先级
- 任务对象生成
- 任务执行页展示
- 推荐提问模板
- 内嵌导师上下文注入
- 任务完成状态记录

#### 第二优先级
- 用户自解释提交
- 微练习记录
- 导师侧结构化总结
- 执行证据落库

#### 第三优先级
- 自动完成质量评估
- 更精细的执行行为埋点
- 多轮动态任务改写
- 更强的导师约束与策略切换

## 18. 适合比赛表达的核心亮点

如果用于比赛展示，任务执行模块最值得讲的不是“我们也接了一个 LLM”，而是以下三点。

### 18.1 我们不让用户面对一个空白聊天框

普通用户面对空白聊天框时，最大问题不是“没有模型”，而是：
- 不知道当前该问什么
- 不知道怎样追问
- 不知道怎样判断自己是否学会

我们的系统为用户补上了这一层脚手架。

### 18.2 我们把高质量使用 LLM 的方法产品化了

系统输出的不是知识大段正文，而是：
- 当前学习框架
- 更有效的提问方式
- 更有效的追问方向
- 更可靠的完成标准与自检方式

这本质上是在教用户：

> **如何把 LLM 用成学习工具，而不是答案生成器。**

### 18.3 我们把学习过程变成了可留痕、可反馈、可调整的闭环

执行过程不会在聊天结束后消失，而是能沉淀为：
- 学习证据
- 当前理解状态
- 新暴露出的阻塞点
- 下一步更合理的动作建议

这才使得“诊断—规划—执行—反馈—调整”真正形成闭环。

---

## 19. 一句话定义任务执行模块

可作为文档摘要或答辩表达：

> **任务执行模块不是知识内容的直接灌输层，而是把规划结果转化为可推进、可对话、可检查、可留痕的学习任务脚手架；系统提供知识框架、提问方式与完成标准，具体知识理解通过用户与受约束的 LLM 导师互动完成。**

---

## 20. 小结

与前三个模块一脉相承，任务执行模块的核心不是“教更多知识”，而是“教用户如何借助 LLM 更好地学习知识”。

从系统视角看，本模块承担了三层关键价值：

1. **把规划结果真正落地成用户能做的具体学习动作**
2. **把高质量提问、追问、自解释与自检方法嵌入执行过程**
3. **把学习行为沉淀为可服务后续反馈与动态调整的证据**

因此，本模块既是整个 AI 学习导航系统中最贴近用户真实学习行为的一层，也是最能体现你们项目核心差异化的一层：

> **我们解决的不是用户“没有大模型可用”，而是用户“不会把大模型用成真正有效的学习工具”。**
