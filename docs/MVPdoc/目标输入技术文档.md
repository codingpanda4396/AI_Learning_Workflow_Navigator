# 目标输入技术文档

## 1. 文档目标

本文档用于明确 AI 学习导航系统中“目标输入（Goal Input）”模块的职责、输入形式、数据结构、接口设计与落库方案，作为后续诊断、规划、任务执行与动态调整链路的起点。

该模块的核心目标不是“收一段用户文本”，而是把用户当前学习意图转化为**可解释、可计算、可流转**的结构化目标对象。

---

## 2. 模块定位

在系统最小闭环中，目标输入位于第一环：

**目标输入 → 用户诊断 → 学习规划 → 任务执行 → 反馈与 Next Action**

目标输入模块负责完成三件事：

1. 收集用户当前想解决的学习问题
2. 收集与该目标相关的上下文约束
3. 产出后续模块可直接消费的结构化目标快照

---

## 3. 设计原则

### 3.1 自然表达优先，结构化补充
用户首先用自然语言表达目标，系统再通过轻量结构化字段补齐规划所需约束。

### 3.2 为后续链路服务，而不是只服务输入页
该模块的产物必须能被后续诊断、规划、任务执行与动态调整模块直接使用。

### 3.3 保留原始意图，避免“系统理解覆盖用户表达”
既要保存用户原话，也要保存系统解析后的结构化结果，方便后续解释、追踪与纠偏。

### 3.4 最小可行，但具备扩展性
先支持最小闭环必需字段，同时预留课程、章节、知识点、资源来源等锚点能力。

---

## 4. 用户要输入什么目标

目标输入不应只是一句“我想学 X”，而应拆为：

### 4.1 核心目标（必填）
回答问题：

**你现在最想解决的学习目标是什么？**

示例：
- 我想搞懂链表
- 我明天考试，想快速复习栈和队列
- 我会图的概念，但不会做最短路径题
- 我想系统学习 408 数据结构

这一层决定用户的主学习意图，是整个模块的主输入。

### 4.2 目标约束（建议补充）
用于缩小规划空间，让后续结果更个性化、更贴合当前状态。

建议采集：
- 时间预算
- 自评基础水平
- 学习偏好
- 学习深度预期
- 紧迫程度
- 学习原因（考试 / 补基础 / 做题 / 项目）

### 4.3 目标锚点（可选）
用于挂接知识图谱、章节结构、题库、课程资源。

可选信息：
- 学科 / 课程
- 章节
- 知识点
- 当前材料来源（教材、PPT、题目、试卷）

---

## 5. 前端输入形式设计

不建议只做一个自由文本框。推荐采用：

**1 个主输入框 + 4 个轻量辅助输入项**

### 5.1 主输入框
字段名：`rawGoalText`

文案建议：

> 你现在最想解决的学习目标是什么？

示例占位文案：
- 我想搞懂链表
- 我明天考试，想快速过一遍树和图
- 我会概念但不会做题，想强化图算法

### 5.2 时间预算
字段名：`timeBudget`

可选值建议：
- `WITHIN_15_MIN`
- `WITHIN_30_MIN`
- `WITHIN_60_MIN`
- `MULTI_DAY`
- `LONG_TERM`

### 5.3 当前自评水平
字段名：`selfReportedLevel`

可选值建议：
- `BEGINNER`
- `BASIC`
- `PARTIAL_UNDERSTANDING`
- `CAN_EXPLAIN_BUT_NOT_APPLY`
- `SOLID_BUT_WANT_IMPROVE`

### 5.4 学习偏好
字段名：`preferenceTags`

可多选，建议值：
- `CONCEPT_FIRST`
- `EXAMPLE_FIRST`
- `PRACTICE_FIRST`
- `LESS_THEORY`
- `STEP_BY_STEP`
- `EXAM_ORIENTED`

### 5.5 学习目标类型（可自动推断，也可显式选择）
字段名：`goalTypeHint`

可选值建议：
- `LEARN_NEW_CONCEPT`
- `REVIEW_FOR_EXAM`
- `FIX_SPECIFIC_BLOCKER`
- `PRACTICE_ENHANCEMENT`
- `BUILD_SYSTEMATIC_UNDERSTANDING`

## 6. 为什么不能只收一句自然语言

如果只有“我想学链表”这一句，会出现以下问题：

### 6.1 无法判断真实意图
“学链表”可能意味着：
- 完全不会，想入门
- 学过但遗忘，想复习
- 会概念不会做题
- 临近考试，想快速突击

这四种情况的规划完全不同。

### 6.2 诊断模块会被迫承担全部理解压力
前置输入信息过少，会导致后续诊断问题过多、过重，增加用户负担。

### 6.3 目标难以在后续流程中被稳定复用
系统需要知道：
- 用户最初想解决什么问题
- 当时是否时间紧迫
- 预期掌握到什么程度
- 后续是否偏离原始目标

这些都要求目标对象结构化保存。

---

## 7. 核心数据分层设计

目标输入后的数据，不应只保存一份，而应分为三层：

1. 原始输入层
2. 结构化目标层
3. 目标上下文快照层

---

## 8. 第一层：原始输入层

保留用户原始表达，方便解释、调试与回溯。

### 8.1 数据对象：LearningGoalInput
用于表示用户本次提交的原始目标输入。

#### 字段设计

| 字段名 | 类型 | 是否必填 | 说明 |
|---|---|---:|---|
| rawGoalText | string | 是 | 用户自然语言输入的原始目标 |
| timeBudget | enum | 否 | 时间预算 |
| selfReportedLevel | enum | 否 | 用户自评基础水平 |
| preferenceTags | string[] | 否 | 偏好标签 |
| goalTypeHint | enum | 否 | 用户显式选择的目标类型提示 |
| subjectHint | string | 否 | 学科 / 课程提示 |
| topicHints | string[] | 否 | 用户明确指定的主题或知识点 |
| sourceContext | string | 否 | 当前材料来源，如教材/试卷/OJ |

#### Java 示例
```java
public class LearningGoalInput {
    private String rawGoalText;
    private TimeBudget timeBudget;
    private SelfReportedLevel selfReportedLevel;
    private List<PreferenceTag> preferenceTags;
    private GoalType goalTypeHint;
    private String subjectHint;
    private List<String> topicHints;
    private String sourceContext;
}
```

---

## 9. 第二层：结构化目标层

由系统根据用户原始输入和辅助字段解析得到，是后续诊断与规划的主消费对象。

### 9.1 数据对象：StructuredLearningGoal

#### 字段设计

| 字段名 | 类型 | 说明 |
|---|---|---|
| rawGoalText | string | 原始目标文本 |
| normalizedGoalText | string | 标准化后的目标描述 |
| goalType | enum | 系统识别的目标类型 |
| subject | enum/string | 所属学科或课程 |
| topicScopeType | enum | 目标范围：单知识点 / 多知识点 / 章节 / 课程级 |
| topics | string[] | 解析出的主题列表 |
| intentDescription | string | 系统理解后的目标意图描述 |
| timeBudget | enum | 时间预算 |
| urgencyLevel | enum | 紧迫程度 |
| expectedDepth | enum | 预期掌握深度 |
| selfReportedLevel | enum | 用户自评水平 |
| preferenceTags | enum[] | 学习偏好标签 |
| constraints | string[] | 解析出的约束信息 |
| sourceContext | string | 材料来源 |

#### Java 示例
```java
public class StructuredLearningGoal {
    private String rawGoalText;
    private String normalizedGoalText;
    private GoalType goalType;
    private String subject;
    private TopicScopeType topicScopeType;
    private List<String> topics;
    private String intentDescription;
    private TimeBudget timeBudget;
    private UrgencyLevel urgencyLevel;
    private ExpectedDepth expectedDepth;
    private SelfReportedLevel selfReportedLevel;
    private List<PreferenceTag> preferenceTags;
    private List<String> constraints;
    private String sourceContext;
}
```

### 9.2 这一层的作用
这层数据主要服务于：
- 诊断问题裁剪
- 规划策略选择
- 首任务粒度控制
- 解释性文案生成
- 后续目标对齐与偏移检测

---

## 10. 第三层：目标上下文快照层

这一层不是用户输入，而是系统基于结构化目标进一步推导出的**规划消费快照**。

### 10.1 数据对象：GoalContextSnapshot

#### 字段设计

| 字段名 | 类型 | 说明 |
|---|---|---|
| structuredGoal | StructuredLearningGoal | 结构化目标对象 |
| requiresDiagnosis | boolean | 是否必须进入诊断 |
| planningMode | enum | 规划模式 |
| entryGranularity | enum | 首任务粒度 |
| strategyHints | string[] | 对规划器的策略提示 |
| riskTags | string[] | 风险标签 |
| explanationFocus | string[] | 后续解释文案的重点 |
| createdFrom | string | 快照来源，如 USER_INPUT_V1 |
| version | int | 快照版本号 |

#### Java 示例
```java
public class GoalContextSnapshot {
    private StructuredLearningGoal structuredGoal;
    private boolean requiresDiagnosis;
    private PlanningMode planningMode;
    private EntryGranularity entryGranularity;
    private List<String> strategyHints;
    private List<String> riskTags;
    private List<String> explanationFocus;
    private String createdFrom;
    private Integer version;
}
```

### 10.2 这一层的典型推导结果
例如用户输入：

> 我明天考试，想快速搞懂栈和队列，最好能会做题

系统可推导出：
- `requiresDiagnosis = true`
- `planningMode = FAST_TRACK`
- `entryGranularity = MICRO`
- `riskTags = [TIME_PRESSURE, SHALLOW_UNDERSTANDING_RISK]`
- `strategyHints = [CORE_CONTRAST_FIRST, ONE_EXAMPLE_ONE_PRACTICE]`

## 12. 解析逻辑建议

目标输入模块不应直接依赖一次自由生成式 LLM 来“决定一切”，而应采用：

**规则抽取 + 受限推断 + 可选 LLM 补充解释**

### 12.1 优先用规则抽取的内容
- 时间约束词：明天考试 / 这周 / 30 分钟
- 紧迫语义：马上 / 快速 / 突击
- 深度语义：搞懂 / 会做题 / 系统掌握
- 主题实体：链表 / 树 / 图 / 最短路径

### 12.2 可由 LLM 辅助的内容
- 目标归类
- 用户意图标准化描述
- 主题补全
- 解释性文本生成

### 12.3 不建议直接交给 LLM 决定的内容
- 是否进入诊断
- 规划模式
- 首任务粒度
- 风险标签定义

这些更适合由规则或受限决策表生成，保证稳定与可控。

## 15. 模块与后续链路的关系

### 15.1 对诊断模块的作用
目标输入决定诊断问题的裁剪范围。

例如：
- 用户目标为“快速考前复习”，诊断应短平快
- 用户目标为“系统掌握”，诊断可更完整

### 15.2 对规划模块的作用
结构化目标决定：
- 候选策略池
- 首任务入口
- 任务粒度
- 解释层文案

### 15.3 对执行模块的作用
执行阶段需要知道：
- 当前任务是在服务什么目标
- 是快速达标还是长期掌握
- 用户偏好是什么

### 15.4 对动态调整模块的作用
后续反馈必须能够相对“初始目标快照”进行比较，才能判断：
- 是否偏离目标
- 是否需要重规划
- 当前 Next Action 是否仍与最初目标一致

## 16. MVP 版本建议

如果当前时间紧，不建议一次做完整版本。推荐先实现最小可用集。

### 16.1 前端 MVP 字段
只保留：
- `rawGoalText`
- `timeBudget`
- `selfReportedLevel`
- `preferenceTags`

### 16.2 后端 MVP 产出
至少解析出：
- `goalType`
- `topics`
- `urgencyLevel`
- `expectedDepth`
- `planningMode`

### 16.3 MVP 已能支撑的能力
- 目标基础结构化
- 诊断裁剪
- 规划首任务分流
- 个性化解释生成
- 后续目标对齐

## 17. 演进路线建议

### V1：目标输入结构化
实现本文件中的原始输入层 + 结构化目标层 + 快照层。

### V2：目标锚点增强
加入课程、章节、知识点、资源来源、教材页码等锚点。

### V3：目标历史与变更
支持目标修订、目标漂移检测、阶段性子目标拆分。

### V4：多轮目标对齐
支持用户在执行过程中的目标变化，例如：
- 从“搞懂概念”切换到“做题强化”
- 从“这周学完”切换到“今晚先过一遍”

---

## 18. 最终结论

目标输入模块本质上不是一个输入框，而是一个**学习目标结构化入口**。

推荐采用如下总方案：

- **输入内容**：核心目标 + 轻量约束 + 可选锚点
- **输入形式**：自然语言主输入 + 少量结构化选项
- **保存结构**：原始输入 `LearningGoalInput` + 结构化目标 `StructuredLearningGoal` + 规划消费快照 `GoalContextSnapshot`

这样设计后，系统后续的诊断、规划、执行与动态调整都能围绕同一个稳定、可解释的目标对象运行，而不是围绕一段模糊文本临时猜测。

## 19. 对应你当前笔记的收敛表达

结合当前讨论，目标输入模块可以收敛为以下三层对象：

### UI 层
让用户输入“现在想朝哪个学习方向推进”，并补充目标约束与偏好。

### DS 层
形成：
- `LearningGoalInput`
- `StructuredLearningGoal`
- `GoalContextSnapshot`

### DB 层
落为：
- `goal_input_record`
- `goal_context_snapshot`

这三层分别对应：
- 用户表达
- 系统理解
- 系统消费

这就是目标输入模块最合理、最稳定的技术形态。
