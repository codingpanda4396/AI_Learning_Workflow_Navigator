# 学习规划技术方案

## 1. 文档目标

本文档用于明确 AI 学习导航系统中“学习规划（Learning Planning）”模块的职责边界、输入输出结构、核心决策链路、个性化规划生成机制、与任务执行模块的协同方式，以及最小可落地的技术实现方案。

该文档与《目标输入技术文档》《用户诊断技术方案》协同工作：

- **目标输入模块**负责把用户当前学习意图转化为结构化目标快照，回答“用户这次想学什么、想学到什么程度、在什么约束下学”
- **用户诊断模块**负责识别当前目标下的用户状态、阻塞点与风险标签，回答“用户现在以什么状态去学、当前最可能卡在哪里”
- **学习规划模块**负责把“目标 + 状态 + 约束”编译为一份可执行、可解释、可调整的学习推进方案，回答“系统接下来应该如何推进用户学习”

因此，学习规划模块不是一个自由生成学习建议的模块，而是系统闭环中的**策略决策层 + 路径编排层 + 执行上游控制层**。

---

## 2. 模块定位

在系统的最小闭环中，学习规划位于第三环：

**目标输入 → 用户诊断 → 学习规划 → 任务执行 → 反馈与 Next Action**

其中，学习规划承担的核心职责不是“给出一段建议”，而是：

1. **消费前两个模块沉淀下来的结构化信息**
2. **根据用户画像做受约束的个性化策略选择**
3. **产出可直接驱动执行模块运行的任务化计划结果**
4. **预埋执行阶段的证据采集点与动态调整规则**

因此，本模块本质上是一个：

> **受约束的个性化规划决策器 + 学习任务编排器**

它既服务“前端展示”，更服务“下游执行”。

---

## 3. 设计目标

### 3.1 核心目标

学习规划模块需要同时满足以下四个目标：

#### 目标 1：规划结果必须真正消费用户画像

规划不能只是复述“你基础较弱”这类诊断结论，而必须让用户画像真正改变以下决策：

- 从哪里开始学
- 先讲概念还是先做题
- 一次安排多少内容
- 每一步任务时长多长
- 是否需要补前置知识
- 是否插入更多检查点
- 是否采用压缩路径或保底路径

也就是说，用户画像的价值不在于“展示出来”，而在于被**编译成规划控制变量**。

#### 目标 2：规划结果必须可执行，而不是停留在建议层

规划输出不应是一大段自然语言，而应是可结构化消费的计划对象，包括：

- 推荐入口点
- 规划策略
- 阶段划分
- 任务序列
- 检查点
- 成功标准
- 反馈采集字段
- 动态调整策略

这样后续执行模块才能直接按计划推进。

#### 目标 3：规划结果必须可解释

系统不仅要给出“学什么”，还要让用户理解：

- 为什么从这里开始
- 为什么先做这一类任务
- 这一步在解决什么问题
- 如果跳过会产生什么风险
- 后面会根据什么信号进行调整

这样规划才具备“懂用户”的体感。

#### 目标 4：规划结果必须可调整

规划不是一次性静态结果，而是当前时刻下的最优推进假设。执行阶段产生的新证据应能反向修正规划假设。

---

### 3.2 非目标

本模块当前阶段不追求以下能力：

- 不追求完全开放式、长周期、强自主的课程编排系统
- 不追求覆盖所有学科的一般化最优策略搜索
- 不追求用 LLM 端到端自由生成整套教学内容
- 不追求复杂的强化学习式规划优化

当前更合理的目标是构建：

> **针对 408 / 计算机学习场景的、基于结构化输入与用户画像的个性化规划原型**

---

## 4. 规划模块在整体系统中的作用

从业务视角看，学习规划模块要解决的是：

> **在当前目标、当前状态、当前约束下，系统到底应该给用户安排一条怎样的推进路径。**

与其他模块的关系如下：

### 4.1 对上承接目标输入模块

规划模块需要消费目标输入模块提供的目标结构化结果，用于确定：

- 学习对象范围
- 学习目标类型
- 目标优先级与紧迫度
- 预期掌握深度
- 时间预算与学习窗口
- 用户偏好的学习方式

这些信息决定规划的**上界与约束**。

### 4.2 对上承接用户诊断模块

规划模块需要消费用户诊断模块提供的用户画像快照，用于确定：

- 用户当前基础层级
- 当前核心阻塞点
- 更适合的学习推进方式
- 当前风险标签
- 建议的任务粒度和反馈频率
- 是否需要前置补足或保底路径

这些信息决定规划的**起点与推进方式**。

### 4.3 对下驱动任务执行模块

规划模块的输出应能直接映射为：

- 执行任务单元
- 每个任务的指令和完成标准
- 执行期间要收集的证据字段
- 阶段切换条件
- 动态调整规则

因此，规划是执行的上游配置，而不是纯展示页内容。

---

## 5. 规划模块的核心输入

学习规划模块不应直接消费前端的原始表单字段，而应消费前两个模块产出的结构化对象。

---

### 5.1 来自目标输入模块的输入

建议主要消费以下对象：

#### 1）StructuredLearningGoal

表示用户本轮学习目标的结构化定义。

典型字段如下：

```java
public class StructuredLearningGoal {
    private String goalId;
    private String rawGoalText;
    private GoalType goalType;
    private GoalScope scope;
    private String subject;
    private String topic;
    private String chapter;
    private String concept;
    private MasteryExpectation masteryExpectation;
    private Integer targetDays;
    private Integer dailyMinutes;
    private UrgencyLevel urgencyLevel;
    private List<LearningPreference> preferences;
}
```

#### 2）GoalContextSnapshot

表示目标输入模块对当前目标的附加解释性信息，用于支持规划解释与约束校验。

典型字段如下：

- 目标是否过大
- 目标是否过于模糊
- 当前目标更偏理解 / 训练 / 冲刺 / 查漏补缺
- 时间资源是否明显不足
- 是否存在范围收缩建议

---

### 5.2 来自用户诊断模块的输入

建议主要消费以下对象：

#### 1）LearnerProfileSnapshot

表示当前目标下的用户状态快照。

典型字段如下：

```java
public class LearnerProfileSnapshot {
    private String diagnosisId;
    private FoundationLevel foundationLevel;
    private ConfidenceLevel confidenceLevel;
    private ComprehensionPattern comprehensionPattern;
    private ExecutionPattern executionPattern;
    private List<RiskTag> riskTags;
    private List<BlockerTag> blockerTags;
    private GranularityPreference suggestedGranularity;
    private FeedbackPreference suggestedFeedbackFrequency;
    private List<PlanningHint> planningHints;
}
```

#### 2）DiagnosisEvidenceSummary

表示本轮画像是基于哪些证据得出的，可用于规划解释层与后续审计。

典型字段如下：

- 哪些问题反映基础薄弱
- 哪些回答反映概念混淆
- 哪些答案反映时间压力大
- 哪些特征提示过度自信或执行风险

---

### 5.3 规划模块内部统一输入对象：PlanningContext

为了保证规划链路干净、稳定、可测试，建议在规划模块入口处统一组装 `PlanningContext`，作为规划模块唯一可信输入对象。

```java
public class PlanningContext {
    private StructuredLearningGoal goal;
    private GoalContextSnapshot goalSnapshot;
    private LearnerProfileSnapshot learnerProfile;
    private DiagnosisEvidenceSummary diagnosisEvidence;

    private PlanningMode planningMode;
    private List<String> constraints;
    private List<String> strategyHints;
    private List<String> riskTags;
}
```

其作用是：

- 隔离前端原始输入与规划逻辑
- 把前两个模块的结果归一化
- 为规则判断、策略选择、LLM 受限增强提供统一上下文

---

## 6. 规划模块如何消费并落实用户画像

这是本模块最关键的问题。

用户画像在规划模块中的价值，不是被展示，而是要真正改变规划决策。也就是说：

> **画像信号必须下沉为规划参数，而不是停留在说明文本。**

---

### 6.1 画像到规划的映射原则

规划模块必须将以下画像信号映射为实际决策：

| 用户画像信号 | 对应规划影响 |
|---|---|
| 基础薄弱 | 起点前移、增加前置补足、缩小任务粒度 |
| 概念混淆严重 | 先走概念澄清而不是直接训练 |
| 题目会做但解释不清 | 插入自我解释/复述任务 |
| 时间很紧 | 优先压缩路径，减少低优先内容 |
| 过度自信风险 | 插入校准型检查点 |
| 执行稳定性差 | 任务更短、反馈更频繁 |
| 有明确考试导向 | 采用考点优先、题型优先路径 |
| 偏好例题驱动 | 路径中提高 example/practice 任务占比 |

因此，规划模块消费画像的本质，是把画像转成以下几类控制变量：

- `recommendedEntry`：推荐入口点
- `strategyCode`：规划主策略
- `taskGranularity`：任务粒度
- `taskMix`：任务类型配比
- `feedbackFrequency`：反馈频率
- `checkpointDensity`：检查点密度
- `fallbackPath`：保底路径 / 回退路径

---

### 6.2 规划中必须体现个性化的关键维度

#### 1）起点个性化

即：从哪里开始学。

同样是“学习链表”，不同用户的推荐入口应不同：

- 完全不会：从节点、指针、连接关系开始
- 学过但混乱：从核心概念对比开始
- 会概念不会做题：从典型题切入
- 考前复习：从高频易错点切入

因此规划对象中需要单独定义 `recommendedEntry`。

#### 2）路径个性化

不是所有用户都走统一路径“概念 → 例题 → 练习 → 总结”。

典型变体包括：

- 基础重建路径
- 概念澄清路径
- 练习驱动路径
- 冲刺压缩路径
- 局部漏洞修补路径

#### 3）粒度个性化

规划必须根据用户状态决定任务切分粒度：

- 基础薄弱、执行不稳：任务更细，单步更短
- 基础较好、状态稳定：任务可适当合并
- 时间受限：任务时长更短，优先形成短闭环

#### 4）任务类型个性化

规划不应固定为单一任务类型，而应组合如下任务：

- `READ_CONCEPT`
- `COMPARE_CONFUSIONS`
- `WATCH_EXAMPLE`
- `SOLVE_PRACTICE`
- `SELF_EXPLAIN`
- `QUIZ_CHECK`
- `ERROR_REFLECTION`

不同画像对应不同任务类型占比。

#### 5）节奏个性化

执行节奏需要由规划预先定义：

- 高风险用户：每步后检查
- 稳定用户：阶段后统一反馈
- 冲刺用户：短反馈、多校准、少长反思

#### 6）解释个性化

用户需要知道：

- 为什么从这里开始
- 这一步是为了解决什么问题
- 跳过会怎样
- 系统将如何根据表现调整后续路径

这部分由模板 + 受限 LLM 共同完成。

---

## 7. 个性化规划的生成机制

规划模块不应采用“让 LLM 直接自由生成完整学习计划”的方式，而应采用分层生成机制：

> **画像归因 → 策略选择 → 规划骨架生成 → 任务化展开 → 解释增强**

这样能兼顾个性化、可控性和可解释性。

---

### 7.1 第一层：策略选择（Strategy Selection）

规划的第一步不是排任务，而是根据目标与画像，确定当前最适合的推进策略。

建议定义一组**有限策略模板**，由规则系统主导选择。

例如：

- `FOUNDATION_REBUILD`：基础重建
- `CONCEPT_CLARIFICATION`：概念澄清
- `PRACTICE_DRIVEN`：练习驱动
- `EXAM_SPRINT`：考前压缩冲刺
- `SYSTEMATIC_PROGRESSIVE`：系统渐进学习
- `PATCH_LOCAL_GAP`：局部漏洞修补

这些策略不是文案标签，而是后续规划骨架的上层控制变量。

---

### 7.2 第二层：规则生成候选策略

策略选择建议以规则为主、LLM 为辅。

典型规则示例：

#### 情况 1：时间紧 + 考试导向 + 基础并非完全不会

- 主策略：`EXAM_SPRINT`
- 次策略：`PATCH_LOCAL_GAP`

#### 情况 2：用户自评“会一点”，但诊断显示概念混淆严重

- 主策略：`CONCEPT_CLARIFICATION`

#### 情况 3：概念能说出，但做题正确率低

- 主策略：`PRACTICE_DRIVEN`

#### 情况 4：目标范围较大、预算较长、适合系统推进

- 主策略：`SYSTEMATIC_PROGRESSIVE`

#### 情况 5：当前主题强依赖前置知识，且前置缺口明显

- 主策略：`FOUNDATION_REBUILD`

这种“规则给候选”的方式，有以下优点：

- 稳定
- 易调试
- 易解释
- 比赛展示时更可信

---

### 7.3 第三层：规划骨架生成（Plan Skeleton Building）

一旦策略选定，系统就不再自由发挥，而是根据策略模板生成规划骨架。

规划骨架至少包括：

- 推荐入口点
- 阶段数量与顺序
- 每阶段核心目标
- 每阶段任务类型组合
- 检查点位置
- 粒度建议
- 预计时长

例如：

#### `FOUNDATION_REBUILD` 骨架

- 阶段 1：基础概念补足
- 阶段 2：最小例题理解
- 阶段 3：低门槛训练
- 阶段 4：快速回顾与校验

#### `PRACTICE_DRIVEN` 骨架

- 阶段 1：题型识别与方法唤醒
- 阶段 2：典型题训练
- 阶段 3：错因归纳
- 阶段 4：变式巩固

#### `EXAM_SPRINT` 骨架

- 阶段 1：高频考点速过
- 阶段 2：高频错误扫雷
- 阶段 3：限时自检
- 阶段 4：保底修补

---

### 7.4 第四层：任务化展开（Task Expansion）

规划骨架生成后，需要将阶段展开为任务级对象，供执行模块直接消费。

任务展开时主要根据以下变量进行控制：

- 任务类型
- 任务数量
- 单任务时长
- 完成标准
- 证据采集字段
- 是否依赖前置任务

例如，某阶段可展开为：

1. 阅读概念说明（8 分钟）
2. 对比两个易混概念（6 分钟）
3. 完成 2 道小练习（10 分钟）
4. 用自己的话解释核心区别（4 分钟）

---

### 7.5 第五层：解释增强（Explanation Enhancement）

规划骨架和任务结构确定后，再引入模板或受限 LLM 做语言层增强，而不是让 LLM 决定底层路径。

LLM 适合负责：

- 生成更自然的任务标题和说明
- 补充 `why this step`
- 生成“跳过风险说明”
- 生成“动态调整说明”
- 将结构化证据转成用户可理解的话术

这样既能保证体感个性化，又不会失控。

---

## 8. 规划结果的数据结构设计

学习规划模块输出的结果，必须同时服务前端展示与后端执行，因此建议以结构化计划对象为核心。

---

### 8.1 顶层计划对象

```java
public class LearningPlan {
    private String planId;
    private String goalId;
    private String diagnosisId;
    private String strategyCode;
    private RecommendedEntry recommendedEntry;
    private List<PlanStage> stages;
    private List<String> riskTags;
    private DynamicAdjustmentPolicy adjustmentPolicy;
    private PlanExplanation explanation;
}
```

字段说明：

- `strategyCode`：本轮采用的主策略
- `recommendedEntry`：推荐起点及原因
- `stages`：阶段化计划结果
- `riskTags`：本轮规划重点关注的风险标签
- `adjustmentPolicy`：执行后如何调整路径
- `explanation`：供前端展示的解释性信息

---

### 8.2 推荐入口点对象

```java
public class RecommendedEntry {
    private String conceptId;
    private String title;
    private Integer estimatedMinutes;
    private String reason;
    private String skipRisk;
}
```

它的核心作用是回答：

- 为什么从这里开始
- 这个入口在解决用户什么问题
- 跳过它会带来什么风险

---

### 8.3 阶段对象

```java
public class PlanStage {
    private String stageCode;
    private String title;
    private String objective;
    private Integer estimatedMinutes;
    private List<PlanTask> tasks;
    private Checkpoint checkpoint;
}
```

阶段是连接“总体规划”和“任务执行”的中间层，便于：

- 做节奏控制
- 做阶段性总结
- 做动态调整切换

---

### 8.4 任务对象

```java
public class PlanTask {
    private String taskId;
    private TaskType taskType;
    private String title;
    private String instruction;
    private Integer estimatedMinutes;
    private String successCriteria;
    private List<String> evidenceToCollect;
    private List<String> dependsOnTaskIds;
}
```

关键字段说明：

- `taskType`：决定执行页面如何组织互动
- `instruction`：任务执行说明
- `successCriteria`：任务完成标准
- `evidenceToCollect`：执行后需要收集的证据字段
- `dependsOnTaskIds`：支持顺序控制与回退逻辑

---

### 8.5 检查点对象

```java
public class Checkpoint {
    private String checkpointType;
    private String title;
    private List<String> checkItems;
    private String passCondition;
    private String failAction;
}
```

检查点的价值在于：

- 校准“用户是否真的掌握”
- 识别规划假设是否成立
- 决定后续是继续前进、回退，还是切换路径

---

### 8.6 动态调整策略对象

```java
public class DynamicAdjustmentPolicy {
    private List<AdjustmentRule> rules;
}

public class AdjustmentRule {
    private String trigger;
    private String action;
    private String reason;
}
```

规则示例：

- 若概念检查连续错误，则回退到概念澄清任务
- 若练习正确率 > 80%，则跳过重复性基础任务
- 若连续两次反馈“太难”，则切换更小粒度任务
- 若时间预算即将耗尽，则自动切换保底路径

---

## 9. 规划结果如何与任务执行有机结合

这是学习规划模块区别于普通“建议生成器”的关键。

> **规划结果不是展示物，而是执行引擎的上游配置。**

---

### 9.1 执行模块直接消费任务单元

任务执行模块不应重新“理解规划文案”，而应直接读取结构化任务对象：

- 任务标题
- 任务说明
- 任务时长
- 任务类型
- 成功标准
- 收集哪些证据

这样可以做到：

- 前后端职责清晰
- 执行逻辑可控
- 状态切换标准明确

---

### 9.2 规划阶段预埋证据采集点

规划模块不应只安排任务，还应声明：

> 完成该任务后，系统需要通过哪些证据判断下一步怎么走。

建议的证据类型包括：

- 是否完成
- 完成用时
- 正确率
- 自评理解度
- 是否能复述
- 错误类型标签
- 是否出现“太难 / 太简单 / 太长”的反馈

这些证据将直接喂给反馈模块和 Next Action 模块。

---

### 9.3 执行模块的本质是验证规划假设

规划模块在输出计划时，实际上隐含了一组假设，例如：

- 用户当前真正的问题是概念混淆
- 先补基础比先刷题更有效
- 该用户当前适合 10 分钟粒度微任务

执行模块不是机械完成任务，而是在真实学习行为中验证这些假设。

因此，系统闭环应是：

**用户画像 → 规划假设 → 任务执行 → 新证据 → 修正画像 / 修正规划**

只有这样，系统才是真正的“学习导航系统”，而不是一次性计划生成工具。

---

## 10. 推荐的技术实现链路

为了保证规划模块稳定、可解释、易落地，建议采用以下分层架构。

---

### 10.1 PlanningContextAssembler

职责：

- 读取目标输入模块产出的结构化目标
- 读取用户诊断模块产出的用户画像快照
- 补充必要的历史学习证据（如已有）
- 归一化为 `PlanningContext`

输出：

- `PlanningContext`

该层重点解决“输入统一化”问题。

---

### 10.2 StrategySelector

职责：

- 根据 `PlanningContext` 中的目标类型、时间预算、风险标签、基础层级等信息
- 用规则系统选出主策略、次策略、推荐入口点、粒度建议、反馈频率建议

输出：

- `PlanningDecision`

示例：

```java
public class PlanningDecision {
    private String primaryStrategy;
    private String secondaryStrategy;
    private String recommendedEntryCode;
    private String taskGranularity;
    private String feedbackFrequency;
    private List<String> selectedRiskControls;
}
```

---

### 10.3 PlanSkeletonBuilder

职责：

- 根据策略模板生成规划骨架
- 决定阶段划分、任务类型组合、检查点位置
- 将高层策略落成可执行结构

输出：

- `PlanSkeleton`

该层解决“路径编排”问题。

---

### 10.4 PlanTaskExpander

职责：

- 将骨架展开为任务对象
- 为每个任务补足指令、时长、成功标准、证据采集字段
- 建立任务依赖关系

输出：

- `LearningPlan`

该层解决“执行对接”问题。

---

### 10.5 PlanExplanationEnhancer

职责：

- 基于结构化证据生成自然语言解释层
- 对任务标题、推荐理由、跳过风险、动态调整说明做增强
- 模式建议为模板优先、LLM 受限补充

输出：

- `PlanExplanation`
- 经增强后的 `LearningPlan`

该层解决“懂用户体感”问题。

---

## 11. LLM 在规划模块中的合理角色

学习规划模块中，LLM 的角色必须受限，不能直接接管底层规划决策。

推荐定位如下：

### 11.1 LLM 适合做的事情

- 将结构化规划骨架转成更自然的解释文案
- 生成更贴合用户目标语境的任务标题
- 生成推荐理由、跳过风险说明、动态调整说明
- 对“为什么这一步适合你”做更自然的表达

### 11.2 LLM 不应直接做的事情

- 不应自由决定规划主策略
- 不应无约束决定任务顺序
- 不应脱离规则系统决定是否补前置知识
- 不应端到端生成整套计划结构

推荐模式为：

> **规则系统负责“做决策”，LLM 负责“做表达”。**

这能够兼顾：

- 稳定性
- 可解释性
- 个性化体感
- 比赛展示可信度

---

## 12. 前后模块的协同接口建议

为了与前两个模块合理协同，建议学习规划模块在接口设计上保持清晰的输入输出边界。

---

### 12.1 输入侧协同

规划模块接口建议以如下字段为核心输入：

```json
{
  "goalId": "g_101",
  "diagnosisId": "d_204",
  "sessionId": "s_3001",
  "preferenceAdjustments": {
    "dailyMinutes": 30,
    "learningStyle": "EXAMPLE_FIRST"
  }
}
```

说明：

- `goalId`：关联目标输入模块结果
- `diagnosisId`：关联用户诊断模块结果
- `sessionId`：关联当前学习会话
- `preferenceAdjustments`：允许用户在规划确认前做少量偏好微调

---

### 12.2 输出侧协同

规划模块输出建议至少包含：

```json
{
  "planId": "p_501",
  "status": "PREVIEW_READY",
  "goal": "理解链表",
  "recommendedEntry": {
    "title": "先理解节点、指针与连接关系",
    "estimatedMinutes": 8,
    "reason": "你当前的主要阻塞不在链表操作本身，而在于对节点连接方式缺乏稳定表征。"
  },
  "recommendedStrategy": {
    "code": "FOUNDATION_REBUILD",
    "label": "先补关键基础，再进入练习"
  },
  "stages": [],
  "adjustmentPolicy": {},
  "explanation": {}
}
```

该输出既可供前端预览，也可在确认后转入执行模块。

## 13. 最小可落地版本（MVP）建议

考虑到项目周期与比赛场景，学习规划模块建议分阶段落地。

---

### 13.1 MVP 必做能力

#### 能力 1：统一规划上下文对象

先打通：

- 目标输入结果读取
- 用户诊断结果读取
- 归一化为 `PlanningContext`

#### 能力 2：规则驱动的有限策略选择

至少支持 4~6 种规划策略：

- 基础重建
- 概念澄清
- 练习驱动
- 冲刺压缩
- 局部修补
- 系统渐进

#### 能力 3：生成结构化计划骨架

至少输出：

- 推荐入口点
- 策略 code
- 2~4 个阶段
- 每阶段任务列表
- 简单检查点

#### 能力 4：预埋执行所需证据字段

每个任务至少定义：

- 完成标志
- 预计时长
- 成功标准
- 自评理解度 / 正确率等证据字段

#### 能力 5：基础解释层

先用模板完成：

- 为什么从这里开始
- 为什么当前适合这条路径
- 后面会如何根据表现调整

---

### 13.2 第二阶段增强能力

后续可逐步增加：

- 历史学习证据参与规划
- 更细颗粒度的风险控制
- 更丰富的任务类型组合
- 规划预览页的个性化叙事增强
- LLM 受限改写 explanation
- 更复杂的动态分支规则

---

## 14. 模块价值总结

学习规划模块在整个系统中的核心价值，不是“帮用户写一份计划”，而是：

> **把目标输入模块给出的“想学什么”，以及用户诊断模块给出的“用户当前是什么状态”，转化成一份真正可执行、可解释、可调整的学习推进方案。**

它的关键价值体现在三个层面：

### 14.1 对用户侧

用户不只是得到建议，而是得到：

- 为什么从这里开始
- 为什么这一步适合自己
- 下一步具体做什么
- 做完之后系统如何判断是否继续推进

### 14.2 对系统侧

规划模块把前两个模块的结构化结果编译成执行控制结构，使得系统真正形成：

**目标 → 诊断 → 规划 → 执行 → 调整**

的闭环。

### 14.3 对比赛展示侧

规划模块能够清晰体现：

- 个性化不是文案，而是策略和路径真的不同
- LLM 不是自由生成，而是在规则约束下做表达增强
- 系统不是一次性建议工具，而是具备动态推进能力的学习导航原型

---

## 15. 结论

学习规划模块本质上是整个 AI 学习导航系统中的“决策编排层”。

它向上承接目标输入与用户诊断的结构化结果，向下输出可执行的学习计划与动态调整策略。其核心不是复述用户画像，而是把画像真正转化为：

- 起点选择
- 策略选择
- 路径安排
- 任务粒度
- 任务类型
- 检查点密度
- 反馈频率
- 动态调整规则

只有做到这一点，系统中的“个性化”才不是表层文案，而是真正落实到了学习推进逻辑中。

最终，本模块与前两个模块的合理协同关系可以概括为：

- **目标输入模块**定义“用户要去哪里”
- **用户诊断模块**定义“用户现在在哪里、最可能卡在哪里”
- **学习规划模块**定义“系统应该如何把用户从这里带到那里”

这也是整个 AI 学习导航系统能够成立的关键。
