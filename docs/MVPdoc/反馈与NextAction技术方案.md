# 反馈与 Next Action 技术方案

## 1. 文档目标

本文档用于明确 AI 学习导航系统中“反馈与 Next Action（Feedback & Next Action）”模块的职责边界、输入来源、学习报告结构、反馈归因机制、下一步决策逻辑，以及与《目标输入技术文档》《用户诊断技术方案》《学习规划技术方案》《任务执行技术方案》的协同关系。

该文档与前四个模块协同工作：

- **目标输入模块**负责把用户当前学习意图转化为结构化目标快照，回答“用户现在想学什么、在什么约束下学”
- **用户诊断模块**负责识别当前目标下的用户状态、阻塞点与风险，回答“用户现在以什么状态去学、最可能卡在哪里”
- **学习规划模块**负责把“目标 + 状态 + 约束”编译为可执行计划，回答“系统接下来应该如何推进”
- **任务执行模块**负责把计划落地为具体学习动作并沉淀执行证据，回答“用户这一轮具体做了什么、学得怎么样”
- **反馈与 Next Action 模块**负责对整轮学习进行归纳、评估、归因与调度，回答“本轮到底产生了什么结果、为什么是这个结果、下一步最合理的动作是什么”

因此，本模块不是一个简单的“课后总结页”，而是系统闭环中的：

> **结果编译层 + 原因归因层 + 下一步调度层**

---

## 2. 模块定位

在系统最小闭环中，反馈与 Next Action 位于第五环：

**目标输入 → 用户诊断 → 学习规划 → 任务执行 → 反馈与 Next Action**

本模块承担的核心职责不是“生成一段总结文案”，而是：

1. **汇总前四个模块沉淀下来的结构化信息与执行证据**
2. **生成一份面向用户可理解、可解释的学习报告**
3. **对本轮结果进行结构化归因，判断是目标达成、部分达成还是未达成**
4. **将本轮结果编译为下一步可执行的决策对象，反馈给规划与执行链路**

换句话说：

- 任务执行回答“用户做了什么”
- 反馈模块回答“这些行为意味着什么”
- Next Action 回答“系统接下来该怎么推”

因此，本模块本质上是：

> **学习闭环中的反馈中枢与动态调度器**

---

## 3. 设计边界

### 3.1 不是单纯的任务完成统计页

本模块不能退化为：
- 完成了几个任务
- 花了多少分钟
- 打了多少分

这些只是原始事实，不足以支撑真正的学习反馈与下一步决策。

### 3.2 不是纯 LLM 自由总结器

不能把所有过程数据直接丢给 LLM，让它自由生成“学习报告”和“下一步建议”。

更合理的方式应是：

**结构化证据汇总 → 规则/受限决策 → 生成报告文案**

这样才能保证：
- 结果稳定
- 决策可解释
- 前后模块口径一致
- 便于比赛展示和工程落地

### 3.3 不是只面向用户展示的 UI 层

本模块既面向用户，也面向系统。

它必须同时产出两类结果：
- **用户可读的学习报告**
- **系统可消费的 Next Action 决策对象**

因此，它不是纯展示模块，而是下一轮规划与执行的输入模块。

### 3.4 是“学习结果编译”为“下一步动作”的关键桥梁

前四个模块产生的是：
- 目标定义
- 用户状态判断
- 规划假设
- 执行证据

本模块要把这些离散信息编译成：
- 当前学习结果
- 结果成因
- 后续最优动作

所以它是最完整体现“闭环”价值的模块。

---

## 4. 与前四个模块的协同关系

反馈与 Next Action 模块不能脱离上下文独立存在，它必须严格消费前四个模块的产物。

### 4.1 对上承接目标输入模块

目标输入模块已经提供了：
- 当前学习目标
- 学习范围与知识锚点
- 时间预算
- 学习偏好
- 紧迫程度
- 预期掌握深度

反馈模块需要用这些信息回答：
- 用户这轮原本想解决什么问题
- 本轮结果是否符合最初目标
- 当前执行是否偏离原始目标
- 下一步是否还应围绕原目标继续推进

### 4.2 对上承接用户诊断模块

用户诊断模块已经提供了：
- 当前基础层级
- 核心阻塞点
- 风险标签
- 粒度建议
- 反馈频率建议
- 规划提示信号

反馈模块需要用这些信息回答：
- 系统最初为什么这样判断用户
- 执行结果是否验证了初始诊断
- 原有风险是否缓解
- 是否出现了新的风险与阻塞点

### 4.3 对上承接学习规划模块

学习规划模块已经提供了：
- 推荐入口点 `recommendedEntry`
- 规划主策略 `strategyCode`
- 任务粒度 `taskGranularity`
- 任务类型配比 `taskMix`
- 检查点密度 `checkpointDensity`
- 反馈频率 `feedbackFrequency`
- 成功标准 `successCriteria`
- 保底路径 / 回退路径 `fallbackPath`

反馈模块需要用这些信息回答：
- 当前执行是否达到了规划预设的成功标准
- 规划中的主策略是否适合当前用户
- 是继续当前路径，还是需要插入补练 / 回退 / 切换策略

### 4.4 对上承接任务执行模块

任务执行模块已经提供了：
- 任务完成记录
- 用户与 LLM 的互动证据
- 用户输出内容与微练习结果
- 任务级检查点结果
- 停滞、跳过、超时、重复追问等行为信号

反馈模块要基于这些证据回答：
- 本轮到底学成了什么
- 哪些地方仍未真正掌握
- 当前问题更偏概念缺口、应用缺口还是表达缺口
- 哪一步最值得作为下一步的切入点

### 4.5 对下反向影响规划与执行模块

反馈模块的输出不应止于“报告展示”，还要反向影响：
- 规划策略是否调整
- 任务粒度是否缩小 / 放大
- 是否插入补练任务
- 是否回退前置知识
- 是否增加检查点密度
- 下一轮执行应从哪个知识点或任务模板开始

因此，本模块是连接“上一轮执行”和“下一轮推进”的闭环枢纽。

## 5. 核心设计原则

### 5.1 先做结构化判断，再生成自然语言报告

报告文案只是展示层，真正重要的是内部结构化结论。应优先产出：
- 本轮结果状态
- 已形成能力
- 未解决问题
- 结果归因
- 下一步动作

然后再把这些结果翻译成用户可读的报告。

### 5.2 报告必须回答“为什么”而不是只回答“发生了什么”

本模块不能只输出事实罗列，而应做出解释：
- 为什么当前只算部分达成
- 为什么还不适合进入下一知识点
- 为什么应该补练而不是重学
- 为什么需要切换策略或缩小粒度

### 5.3 决策必须由证据驱动，而不是由单一分数驱动

本模块不建议只依赖“掌握度百分比”做下一步判断，而应综合以下信号：
- 目标达成信号
- 学习质量信号
- 行为质量信号
- 风险变化信号
- 策略匹配信号

### 5.4 反馈既面向用户理解，也面向系统调度

模块输出必须同时支持：
- **用户侧解释与激励**
- **系统侧下一步调度与计划修正**

### 5.5 优先做轻量连续调整，而不是频繁重做整套规划

大多数情况下，反馈模块不需要重新生成整套学习计划，而应优先做：
- 保持主路径
- 插入补练
- 缩小粒度
- 加密检查点
- 回补前置知识

这更符合“导航系统”而不是“反复开新课”的产品定位。

---

## 6. 本模块要解决的核心问题

本模块本质上要回答以下三个问题：

### 6.1 前面的所有页面有哪些信息可以生成最终学习报告？

也就是说，要明确“本轮学习结果”到底由哪些结构化信息与证据共同构成。

### 6.2 学习报告以什么结构呈现？

也就是说，要明确一份真正有产品价值的报告应如何组织，才能让用户看懂自己的学习进展、问题与下一步。

### 6.3 如何反馈并影响下一步决策？

也就是说，要明确反馈结果如何从“报告文案”进入“结构化决策”，并反向作用于规划与执行链路。

---

## 7. 学习报告的数据来源设计

学习报告不能只来自任务执行页，而应消费整个链路的结构化信息。建议将输入来源分为四层。

### 7.1 第一层：目标输入信息 —— 作为报告的起点基线

目标输入模块提供的是“本轮原始目标与约束”，用于定义报告中的起点。

建议消费字段：
- `rawGoalText`
- `goalType`
- `subject / topic / chapter / concept`
- `timeBudget / dailyMinutes / targetDays`
- `selfReportedLevel`
- `preferenceTags`
- `urgencyLevel`
- `masteryExpectation / expectedDepth`
- `sourceContext`

在报告中的主要作用：
- 呈现“这轮原本想解决什么问题”
- 呈现“当时有哪些约束条件”
- 为最终结果评价提供目标参照系

### 7.2 第二层：用户诊断信息 —— 作为报告的状态基线

用户诊断模块提供的是“本轮学习前的用户状态快照”，用于定义系统最初的判断依据。

建议消费字段：
- `LearnerProfileSnapshot`
- `foundationLevel`
- `confidenceLevel`
- `comprehensionPattern`
- `executionPattern`
- `blockerTags`
- `riskTags`
- `suggestedGranularity`
- `suggestedFeedbackFrequency`
- `planningHints`
- `DiagnosisEvidenceSummary`

在报告中的主要作用：
- 呈现“用户一开始处于什么状态”
- 解释“系统为什么选择这套规划方式”
- 作为对比基线，判断后续风险是否缓解、阻塞是否变化

### 7.3 第三层：学习规划信息 —— 作为报告的决策基线

学习规划模块提供的是“系统原本的推进假设”，用于判断本轮执行是否验证了这套假设。

建议消费字段：
- `recommendedEntry`
- `recommendedStrategy.code`
- `whyThisStep`
- `keyEvidence`
- `taskGranularity`
- `taskMix`
- `checkpointDensity`
- `feedbackFrequency`
- `fallbackPath`
- `successCriteria`
- `milestones / stagePlan`

在报告中的主要作用：
- 呈现“系统为什么从这里开始”
- 说明“本轮原本要经过哪些阶段和检查点”
- 判断“当前结果是否达到预期标准”

### 7.4 第四层：任务执行信息 —— 作为报告的真实过程证据

任务执行模块提供的是“本轮真正发生了什么”，这是反馈模块的核心证据层。

建议分为五类证据：

#### 7.4.1 任务完成证据
- 完成了哪些任务
- 每个任务是否完成
- 实际耗时
- 是否存在中断 / 跳过 / 超时
- 每个任务的完成质量等级

#### 7.4.2 对话行为证据
- 是否实际与 LLM 导师发生交互
- 提问轮次
- 是否使用了系统推荐提问脚手架
- 是否有追问、比较、复述、纠错行为
- 是否过度索取直接答案

#### 7.4.3 输出质量证据
- 是否提交了自我解释
- 微练习是否通过
- 是否暴露关键概念错误
- 是否能做基础迁移
- 是否存在“会说不会做”或“会做说不清”

#### 7.4.4 风险事件证据
- 哪些任务停留过久
- 哪些概念被反复追问
- 哪些点被多次答错
- 是否出现敷衍完成、明显疲劳、过度自信等信号

#### 7.4.5 阶段结果证据
- 当前阶段是否达标
- 是继续、补练、回退还是切换策略
- 支持该结论的证据有哪些

## 8. 反馈模块内部统一输入对象设计

为了让“学习报告生成”和“Next Action 决策”使用同一套输入口径，建议在反馈模块入口统一组装 `LearningFeedbackContext`。

```java
public class LearningFeedbackContext {
    private StructuredLearningGoal goal;
    private GoalContextSnapshot goalSnapshot;

    private LearnerProfileSnapshot learnerProfile;
    private DiagnosisEvidenceSummary diagnosisEvidence;

    private LearningPlan plan;
    private PlanExplanationSummary planExplanation;

    private List<TaskExecutionRecord> taskExecutions;
    private List<DialogueEvidence> dialogueEvidences;
    private List<OutputEvidence> outputEvidences;
    private List<CheckpointResult> checkpointResults;

    private LearningProgressSummary progressSummary;
    private List<RiskEvent> riskEvents;
}
```

其作用是：
- 隔离前四个模块的细节差异
- 保证报告与决策使用同一份上下文
- 避免“展示一套口径，决策另一套口径”的问题
- 便于后续做规则判断、模板生成与测试验证

## 9. 学习报告的结构设计

学习报告不应写成流水账，也不能只是“完成情况 + 鼓励语”。

建议采用 **6 段式报告结构**，从“目标 → 判断 → 执行 → 结果 → 归因 → 下一步”形成闭环。

### 9.1 第一段：本轮学习概览

回答问题：**这次学了什么、学了多久、整体结果如何。**

建议展示字段：
- 本轮目标
- 学习主题 / 知识点
- 总耗时
- 完成任务数 / 总任务数
- 当前整体状态：达标 / 部分达标 / 未达标
- 当前建议：继续推进 / 补练 / 回退 / 切换路径

这一段应作为报告头部总览卡。

### 9.2 第二段：起点与系统初始判断

回答问题：**系统一开始为什么这样安排。**

建议展示字段：
- 用户原始目标
- 初始画像摘要
- 主要阻塞点
- 初始风险标签
- 规划起点与策略原因

这一段的价值在于让用户理解：

> 本轮路径不是系统随便安排的，而是基于用户目标与状态做出的起始决策。

### 9.3 第三段：执行过程回顾

回答问题：**用户这轮具体做了什么。**

建议按任务或阶段组织：
- 任务目标
- 是否完成
- 关键行为
- 关键证据
- 是否出现停滞 / 反复追问 / 错误暴露

这里不能只列任务清单，而要强调“学习行为”。

### 9.4 第四段：本轮进展评估

回答问题：**用户到底进步了什么，还差什么。**

建议拆为两个子区块：

#### 9.4.1 已形成的进展
例如：
- 已能用自己的话解释某概念
- 已具备进入下一小步的基础
- 已完成最小迁移应用

#### 9.4.2 仍存在的问题
例如：
- 对关键步骤理解仍不稳定
- 能看懂例子但不能独立迁移
- 容易忽略边界情况

这部分建议优先输出能力性结论，而不是抽象百分比。

### 9.5 第五段：结果归因

回答问题：**为什么本轮会得到这样的结果。**

建议从四个角度归因：
- **目标匹配度**：本轮任务是否真正服务于当前目标
- **策略匹配度**：当前规划策略是否适合用户
- **执行质量**：用户是否完成了足够有效的学习动作
- **知识阻塞点**：当前真正卡住用户的点是什么

这一段是报告区别于普通总结页的关键价值层。

### 9.6 第六段：下一步建议（Next Action）

回答问题：**现在最合理的下一步是什么，为什么。**

建议展示字段：
- 推荐下一步动作
- 推荐理由
- 预计耗时
- 成功标准
- 风险提示

这一段既是报告收束，也是下一轮流程的起点。

## 10. 学习报告的前端展示结构建议

建议前端不要把学习报告做成一大段长文，而应采用“卡片 + 分段解释”的结构：

1. **顶部概览卡**：目标、耗时、完成情况、当前状态
2. **起点判断卡**：初始画像、策略原因、主要风险
3. **执行回顾时间线**：按任务或阶段展示关键事件
4. **进展与问题双栏卡**：已形成能力 / 仍存在缺口
5. **归因解释卡**：为什么是这个结果
6. **Next Action 行动卡**：下一步做什么、为什么、预计多久

这样可以避免报告“像作文”，也更符合产品化展示。

---

## 11. 反馈如何影响下一步决策

这里是本模块最核心的地方。

反馈不应停留在报告展示，而必须进入系统的结构化决策链路。换句话说：

> 报告是给用户看的，Next Action 决策是给系统用的。

因此，本模块必须同时产出：
- `LearningReportView`：用户可读结果
- `NextActionDecision`：系统可消费结果

### 11.1 不要从自然语言报告反推 Next Action

不建议先生成一段报告，再从报告中推测下一步。

正确方向应是：

**先得到结构化决策，再生成报告文案。**

这样可以保证：
- 决策逻辑稳定
- 报告与系统动作一致
- 便于做规则回归测试

### 11.2 Next Action 决策对象设计

建议定义 `NextActionDecision` 作为反馈模块的核心输出对象。

```java
public class NextActionDecision {
    private NextActionType actionType;
    private String targetConceptId;
    private String targetTaskTemplateId;

    private String reasonCode;
    private List<String> supportingEvidence;

    private AdjustmentDirection adjustmentDirection;
    private String strategyDelta;
    private String granularityDelta;
    private String feedbackFrequencyDelta;

    private List<String> newRiskTags;
    private List<String> resolvedRiskTags;

    private Integer estimatedMinutes;
    private List<String> successCriteria;
}
```

其作用是：
- 为后续规划与执行模块提供统一控制信号
- 将“学习结果”翻译为“系统动作”
- 保证决策过程可解释、可审计、可回放

---

## 12. MVP 阶段建议支持的 Next Action 类型

建议 MVP 先支持 6 类动作，足以形成闭环。

### 12.1 CONTINUE

继续当前路径，进入下一个任务或下一个知识点。

适用条件：
- 当前阶段已达标
- 无关键阻塞
- 当前策略匹配度较高

### 12.2 REINFORCE

不换知识点，但插入一小步补练。

适用条件：
- 概念基础已建立
- 但表达或应用还不稳定
- 通过 1~2 个补强任务即可稳住

### 12.3 REMEDIATE_PREREQUISITE

回补前置概念。

适用条件：
- 当前卡点来自前置基础缺失
- 继续当前内容边际收益很低

### 12.4 REDUCE_GRANULARITY

不换方向，但把任务切得更小。

适用条件：
- 用户不是完全不会，而是一步跨度太大
- 当前执行负荷过高
- 风险在于压垮，而不是方向错

### 12.5 CHANGE_STRATEGY

切换推进方式。

例如：
- 概念优先 → 例子优先
- 讲解优先 → 练习优先
- 长任务 → 微任务高反馈

适用条件：
- 内容方向没错
- 但当前方法与用户不匹配

### 12.6 PAUSE_AND_SUMMARIZE

暂停推进，先做阶段总结或收束。

适用条件：
- 时间已到
- 信息量过大
- 继续推进边际收益下降

---

## 13. Next Action 决策的证据驱动机制

Next Action 不应由单一分数驱动，而应由“证据组合”驱动。建议至少综合以下五类信号。

### 13.1 目标达成信号
- 是否达到本轮最小成功标准
- 当前阶段是否完成
- 用户是否偏离原始目标

### 13.2 学习质量信号
- 是否完成了主动解释
- 是否具备基本迁移应用
- 是否仍存在关键误解

### 13.3 行为质量信号
- 是否真实投入
- 是否过度依赖直接答案
- 是否有持续追问与修正

### 13.4 风险变化信号
- 原有风险是否缓解
- 是否出现新风险
- 哪类风险对下一步最敏感

### 13.5 策略匹配信号
- 当前策略是否有效
- 当前粒度是否合适
- 当前反馈频率是否合适

## 14. 反馈结果如何回写到后续规划

反馈模块不一定每次都重新生成整套学习计划，更合理的是输出一个“规划修正补丁”。

建议定义 `PlanAdjustmentPatch`：

```java
public class PlanAdjustmentPatch {
    private boolean keepCurrentStrategy;
    private String newStrategyCode;

    private boolean keepGranularity;
    private String newGranularity;

    private boolean keepCheckpointDensity;
    private String newCheckpointDensity;

    private boolean keepCurrentPath;
    private String redirectConceptId;

    private List<String> insertedTaskTemplates;
    private List<String> removedTaskIds;
}
```

其作用是：
- 允许在现有路径上做轻量增量修正
- 避免反复重建整套规划
- 让系统更像“导航修正”而不是“重新开课”

典型场景：
- 保持主路径，但插入 1 个补练任务
- 不换知识点，但粒度从 `MEDIUM` 改为 `SMALL`
- 不重做规划，只把检查点密度提高一级
- 回退到某个前置概念，再重新进入当前路径

---

## 15. 反馈模块的总体处理链路

建议反馈模块采用以下链路：

**执行证据收集 → 阶段结果判断 → 原因归因 → 生成 NextActionDecision → 生成 LearningReportView → 回写 PlanAdjustmentPatch**

也就是：

**Evidence → Evaluation → Attribution → Decision → Report → Patch**

这条链路的好处是：
- 用户可读报告与系统动作同源
- 结构化判断先行，文案后置
- 便于后续做测试、日志与可解释展示

---

## 16. 核心输出对象设计

建议反馈与 Next Action 模块最终至少输出三类对象。

### 16.1 学习报告对象：LearningReportView

用于前端展示。

建议包含：
- 本轮概览
- 起点与初始判断
- 执行回顾
- 进展与未解决问题
- 结果归因
- 下一步建议

### 16.2 决策对象：NextActionDecision

用于系统做下一步推进。

建议包含：
- 动作类型
- 目标知识点 / 任务模板
- 决策原因码
- 支撑证据
- 策略调整信号
- 风险变化
- 预计耗时
- 成功标准

### 16.3 计划修正对象：PlanAdjustmentPatch

用于规划与执行链路的轻量调整。

建议包含：
- 是否切换策略
- 是否切换粒度
- 是否增加检查点密度
- 是否回退前置知识
- 是否插入 / 删除任务模板

---

## 17. 最小可落地实现建议

从比赛原型角度，建议 MVP 先做以下能力：

### 17.1 先做结构化结果，不强求复杂智能评分

优先把：
- 任务完成情况
- 微练习结果
- 用户自解释质量
- 停滞 / 重复追问 / 跳过行为
- 阶段成功标准达成情况

这些结构化信号先落下来。

### 17.2 先支持有限的结果状态与动作类型

结果状态建议先支持：
- `ACHIEVED`
- `PARTIALLY_ACHIEVED`
- `NOT_ACHIEVED`

动作类型建议先支持：
- `CONTINUE`
- `REINFORCE`
- `REMEDIATE_PREREQUISITE`
- `REDUCE_GRANULARITY`
- `CHANGE_STRATEGY`

### 17.3 先用规则归因 + 模板报告

报告生成不必一开始就完全依赖 LLM。建议采用：

**结构化结论 + 模板拼装 + 受限 LLM 润色（可选）**

这样更稳，也更适合比赛展示“系统逻辑清晰”。

### 17.4 优先保证闭环可跑通

MVP 最重要的不是报告语言多华丽，而是确保：
- 本轮执行能形成报告
- 报告能产出 Next Action
- Next Action 能反向影响下一步执行

只要这三点打通，这个模块就已经成立。

---

## 18. 模块一句话定义

建议将“反馈与 Next Action 模块”的定位定义为：

> **基于目标、诊断、规划与执行证据，生成可解释的学习结果报告，并将学习结果编译为下一步可执行决策的闭环调度模块。**

它回答的不是“用户学完了没有”，而是：
- **学到了什么**
- **为什么学成这样**
- **系统接下来该怎么推**

这正是整个 AI 学习导航系统从“流程串联”走向“真正闭环”的关键一步。
