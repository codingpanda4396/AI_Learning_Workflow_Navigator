# 业务梳理

## 1. 产品本质

这个项目的业务本质不是“让用户跟 AI 聊天”，而是“把一次 AI 辅助学习过程变成可执行、可校验、可复盘的工作流”。

它解决的问题是：

- 用户有学习目标，但不会拆
- 用户知道想学什么，但不知道现在卡在哪
- 用户能得到答案，但不能稳定推进学习
- 用户完成一轮任务后，不知道下一步该继续、巩固还是重规划

因此当前产品主张是：

- 用目标澄清定义起点
- 用诊断识别当前状态
- 用规划给出路径和先后顺序
- 用执行脚手架推进单个任务
- 用报告沉淀证据与下一步动作

## 2. 核心业务闭环

当前系统的完整业务主线是：

1. 用户选择学习主题，创建目标
2. 系统发起诊断，判断基础、阻塞点、推进节奏
3. 系统根据目标和诊断结果生成计划预览
4. 用户确认计划，进入学习会话
5. 系统分配当前任务，驱动用户执行
6. 用户完成任务后，系统更新会话进度
7. 全部任务完成后生成学习报告
8. 系统给出下一步建议，用户确认后形成下一轮入口

这是一条标准的“目标 -> 诊断 -> 规划 -> 执行 -> 报告”闭环。

## 3. 业务对象关系

从业务角度，当前最关键的对象链路如下：

- `Goal`
  用户本轮想学什么
- `Diagnosis Session`
  系统如何判断用户当前所处状态
- `Learner Profile`
  用户当前能力和策略画像
- `Plan`
  系统建议怎么学、先学什么
- `Learning Session`
  一次完整学习执行会话
- `Session Task`
  会话内的具体任务序列
- `Task Runtime`
  某个任务执行过程中的状态与证据
- `Report`
  本轮学习结果总结
- `Next Action`
  下一轮动作建议

关系可以简单理解为：

`一个目标 -> 触发一次诊断 -> 生成一个计划 -> 启动一个学习会话 -> 包含多个任务 -> 产出一份报告 -> 指向下一步动作`

## 4. 各阶段业务目的

### 4.1 目标阶段

业务目的不是收集一句“我要学 XXX”，而是把模糊学习意图转成可驱动后续决策的结构化目标。

当前目标阶段输出了两类关键资产：

- `StructuredLearningGoal`
- `GoalContextSnapshot`

这意味着后续诊断和规划不是基于自由文本直接做，而是基于结构化结果做。

### 4.2 诊断阶段

诊断阶段的业务目的，是识别当前最适合的切入方式，而不是做全面测评。

它主要回答三个问题：

- 用户基础处于什么层级
- 当前最主要的阻塞点是什么
- 适合怎样的推进节奏

产出包括：

- `LearnerProfileSnapshot`
- `DiagnosisEvidenceSummary`
- `LearnerStrategyProfile`

这里的价值在于把“用户不会学”拆成能驱动规划的业务信号。

### 4.3 规划阶段

规划阶段的业务目的，不是生成一堆学习建议，而是给出：

- 为什么从这里开始
- 当前先做哪类任务
- 成功标准是什么
- 风险点是什么

当前规划系统会选择策略，例如：

- 概念澄清
- 框架搭建
- 基础修补
- 冲刺纠偏
- 刷题强化
- 局部修补

也就是说，规划本质上是在决定“学习路径的入口策略”。

### 4.4 执行阶段

执行阶段是本项目最核心的业务舞台。

它不是简单让用户提交答案，而是围绕单任务推进一套受控学习动作：

- 先理解任务目标
- 再展开探索表达
- 再做自我解释
- 再做微检查
- 通过后再收束总结

这和普通聊天式学习的最大区别在于：

- 系统关注的是学习过程是否完整
- 系统会判断是否进入下一状态
- 系统会记录证据，而不是只展示回复

### 4.5 专题脚手架阶段

针对 DFS/BFS 这类知识包，系统额外提供了更强的脚手架。

当前专题业务逻辑是：

- 先帮用户建立结构骨架
- 再验证机制理解
- 再进行训练
- 最后反思提炼

这说明系统正在从“通用流程产品”走向“带知识包编排能力的学习系统”。

### 4.6 报告阶段

报告阶段业务目的不是展示一个完成/未完成，而是沉淀：

- 本轮做成了什么
- 还缺什么
- 学习方法上暴露了什么问题
- 下一轮最适合怎么接

这里已经不只是结果展示，而是下一轮学习入口决策。

## 5. 当前系统真正强调的业务差异

结合代码可以看出，当前系统的差异化不在“AI 回答得更聪明”，而在下面几件事：

### 5.1 学习流程被产品化

不是开放式聊天，而是强流程引导。

### 5.2 学习状态被结构化

目标、诊断、计划、任务状态、报告都不是纯文本，而是明确对象。

### 5.3 执行过程被追踪

系统会记录：

- 消息
- 状态迁移
- 自我解释结果
- checkpoint 结果
- 方法画像

### 5.4 下一步由证据驱动

报告里不是简单让用户“自己想下一步”，而是系统基于证据给出：

- 继续
- 巩固
- 调整策略
- 重规划

## 6. 当前主要业务状态流

### 6.1 会话级状态流

一轮学习会话大致经历：

- `诊断准备`
- `诊断完成`
- `计划预览`
- `计划提交`
- `学习中`
- `学习完成`
- `报告确认下一步`

### 6.2 任务级状态流

单个任务执行时，重点状态包括：

- `ORIENT`
- `EXPLORE`
- `SELF_EXPLAIN`
- `CHECK`
- `REMEDIAL`
- `PASS`

这套状态流体现的是“先暴露理解，再验证理解，再确认达成”。

### 6.3 专题脚手架状态流

DFS/BFS 专题下还有额外状态：

- `STRUCTURE`
- `UNDERSTANDING`
- `TRAINING`
- `REFLECTION`

这是任务内更细的一层学习编排。

## 7. 关键业务输入输出

### 7.1 用户输入

用户在当前系统中主要输入的不是长对话，而是阶段性学习信号：

- 学习主题选择
- 诊断选项
- 任务表达
- 自我解释
- checkpoint 回答
- 收束总结
- 下一步动作确认

### 7.2 系统输出

系统输出的也不是单一文本，而是组合型指导：

- 结构化目标
- 用户画像
- 计划与阶段路径
- 当前任务与提示
- 导师反馈
- 骨架脚手架
- 报告与下一步建议

## 8. 当前业务成熟度判断

从代码结构看，当前项目已经具备比较完整的业务骨架：

- 主流程闭环完整
- 任务执行状态机完整
- 数据模型完整
- 专题脚手架已有成型方向

同时也能看到当前业务重心非常明确：

- 重点不是扩很多页面
- 重点不是做通用对话
- 重点是把“任务执行页”做成真正的学习工作台

## 9. 后续业务演进最可能的方向

基于当前实现，后续业务更可能沿这几条线继续深化：

- 扩更多知识包，而不只是 DFS/BFS
- 让计划与任务生成更贴合用户画像
- 强化报告里的成长沉淀与长期追踪
- 让下一步建议真正能触发下一轮学习闭环

## 10. 一句话总结

当前项目的业务可以概括为：

这是一个把 AI 学习过程拆成可诊断、可规划、可执行、可复盘步骤的学习工作流系统，而任务执行页与专题脚手架引擎，是它最核心的业务展示面。
