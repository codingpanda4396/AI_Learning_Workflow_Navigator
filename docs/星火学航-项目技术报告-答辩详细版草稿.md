# 一、前端技术实现（重点讲清「产品是怎么做出来的」）

## 1. 技术栈与工程结构

### 技术栈

- **框架**：Vue 3（`<script setup>` 组合式 API）
- **构建**：Vite 8
- **路由**：Vue Router 5（`createWebHistory`）
- **状态**：Pinia（工作流与会话 ID、规划预览等）
- **HTTP**：Axios（统一 `GlobalResponse` 解包，见 `frontend/src/api/request.ts`）
- **样式**：Tailwind CSS 3 + 项目内设计 token / 全局样式（`frontend/src/design/`、`style.css`）

### 工程目录（按实际结构归纳）

- `frontend/src/views/`：页面级视图（目标、诊断、规划决策、任务执行、报告、认证等）
- `frontend/src/components/`：可复用组件；执行相关大量集中在 `components/task-run/`（含 `DfsBfs*` 四阶段子组件）
- `frontend/src/api/`：与后端 `/api/*` 一一对应的封装（`goals.ts`、`diagnosis.ts`、`learning-plan.ts`、`task.ts`、`learningScaffold.ts`、`tutor.ts`、`session.ts` 等）
- `frontend/src/stores/`：`workflow`（goal/diagnosis/session/plan 等）、`auth`、`toast`
- `frontend/src/composables/`：`useLearningScaffoldEngine`（脚手架阶段加载/提交）、`useTaskRunSession`（拉取当前任务并与路由对齐）
- `frontend/src/types/`：DTO 与执行台类型（`scaffoldEngine.ts`、`executionWorkbench.ts` 等）
- `frontend/src/constants/`：如 `dfsBfsExecutionConfig.ts`（执行页文案、阶段目标、脚手架按钮配置）、`learningScaffoldPack.ts`

### 模块划分

- **页面**：路由在 `frontend/src/router/index.ts` 定义；`meta.step` 用于鉴权与前置数据校验（如无 `goalId` 不能进诊断等）。
- **组件**：执行页采用 `ExecutionWorkbenchPage` → `PhaseInteractionHost` 按 `phase` 切换四个子阶段组件。
- **状态**：跨步骤持久化用 `sessionStorage`（`workflow` store 内 `STORAGE_KEYS`），当前任务 ID 等由后端 `flow-state` / `current-task` 与前端路由同步。

## 2. 核心页面设计

### 首页（目标输入）— 路由 `/goal`

- **UI 分区**：顶栏 `AppTopBar`；主区 Hero（标题、副标题、流程步骤条）；学科/主题卡片选择；目标表单（含时间、水平等）；底部「继续学习」侧卡（已登录用户展示最近会话入口）。
- **用户路径**：选择学科与主题 → 填写目标文本与约束 → 提交创建目标。
- **接口**：`POST /api/goals`，请求体类型 `CreateGoalRequest`（`rawGoalText` 必填，另有 `timeBudget`、`selfReportedLevel`、`topicHints` 等），响应 `CreateGoalData`（含 `goalId`、`structuredGoal`、`goalContextSnapshot`）。前端将 `goalId` 等写入 Pinia 并持久化。

### 诊断页 — 路由 `/diagnosis`

- **UI 分区**：`WorkflowPageScaffold` 包裹；展示当前主题标签；多组 `FormCard` 单选题；提交区与进度提示。
- **用户路径**：进入页先基于 `goalId` 拉取诊断会话；逐题选择 → 提交答案。
- **接口**：
  - `POST /api/diagnosis/sessions`，body `{ goalId }` → `DiagnosisSessionData`（含 `diagnosisId`、`questions` 等）
  - `POST /api/diagnosis/submissions`，body `{ diagnosisId, answers }` → `SubmitDiagnosisData`（含学习者画像、证据摘要等，供规划页展示）

### 学习规划页 — 路由 `/plan`

- **实现说明**：`LearningPlanView.vue` 仅包装 `LearningPlanDecisionView.vue`。
- **UI 分区**：Hero（`PlanDecisionHero`）、「为什么这样规划」Accordion（`PlanWhyAccordion`）、左侧阶段路径（`PlanStageRail`）、右侧首个任务卡（`PlanFirstTaskCard`）。
- **用户路径**：自动拉取预览 → 阅读理由与路径 → 点击开始执行（提交规划并进入会话）。
- **接口**：
  - `POST /api/learning-plans/preview`，body `{ goalId, diagnosisId }` → `PlanPreviewData`（`planId`、`planId`、`knowledgeKey`/`packId`、`stages`、`tasks` 等）
  - `POST /api/learning-plans/commit`，body `{ planId }` → `CommitPlanData`（`sessionId`、`taskSequence`、`currentTaskId` 等）

### 执行页（核心）— 路由 `/tasks/:taskId/run`（及兼容 `/task`、`/execution`）

**实际承载 UI 的是 `TaskRunView.vue`**；`/execution` 对应 `ExecutionView.vue` 在 `onMounted` 中根据 `sessionId`/`currentTaskId` **重定向**到 `taskRun` 或 `/plan`，本身不是主工作台。

- **UI 分区**：
  - `ExecutionWorkbenchPage`：`ExecutionShell` + 顶栏 `ExecutionTopHeader`（主题名、阶段、阶段目标）+ `PhaseInteractionHost`（按 phase 渲染四阶段之一）
  - 底部 `BottomActionBar`：主按钮文案随阶段变化（如「进入下一阶段」），在反思阶段用于收尾并进入报告
- **用户路径**：
  1. 进入时 `useTaskRunSession` 调用 `GET /api/sessions/{sessionId}/current-task` 拉取当前任务元数据（知识点、`currentStage`、进度条等）
  2. 若 `supportsLearningScaffoldEngine(packId)` 为真（见常量 `LEARNING_SCAFFOLD_ENGINE_PACK_IDS`，当前代码仅包含 **`ds_dfs_bfs`**），则 `useLearningScaffoldEngine` 拉取脚手架阶段
  3. 用户按 `phase` 路由查询串（`?phase=structure|understanding|training|reflection`）与后端 `stageKey` 对齐，完成四阶段后，在反思阶段通过主按钮调用任务完成接口并跳转报告或下一任务
- **主要接口组合**（与阶段强相关，见下节展开）：
  - `GET /api/tasks/{taskId}/scaffold?sessionId=&stage=&workbenchMode=`
  - `GET /api/tasks/{taskId}/learning-scaffold/stage` 或 `.../stage/current`
  - `POST /api/tasks/{taskId}/learning-scaffold/action` 及若干 `.../structure/complete`、`.../conversation/complete` 等
  - `POST /api/ai-tutor/chat` 或 **SSE** `POST /api/ai-tutor/chat/stream`（理解/训练阶段对话）
  - `POST /api/tasks/{taskId}/complete`（任务收口）

### 报告页 — 路由 `/report`

- **已实现**：`ReportView.vue` 拉取 `GET /api/sessions/{sessionId}/report`，展示 Hero、本轮要点、缺口与后续建议等；若尚未完成全部任务会提示「报告暂未生成」并引导回执行页。
- **接口**：`GET /api/sessions/{sessionId}/report`；可选 `GET /api/sessions/{sessionId}/flow-state` 用于前端判断 `reportReady` 与当前任务 ID（路由守卫与报告页均有使用）。

## 3. 执行页（核心创新重点展开）

### 四阶段在前端的体现

| 前端 `PhaseKey`（URL query） | 后端 stageKey（代码中常见大写枚举） | 对应组件 |
|------------------------------|--------------------------------------|----------|
| `structure` | `STRUCTURE` | `DfsBfsStructurePhase` |
| `understanding` | `UNDERSTANDING` | `DfsBfsUnderstandingPhase` |
| `training` | `TRAINING` | `DfsBfsTrainingPhase` |
| `reflection` | `REFLECTION` | `DfsBfsReflectionPhase` |

**说明**：组件与常量命名表明，当前仓库中**与脚手架引擎深度联动的完整四阶段工作台实现**绑定在 **DFS/BFS 对比** 知识点包（`packId === 'ds_dfs_bfs'`）上；其他 `packId` 不会走同一套 `useLearningScaffoldEngine` 分支（`TaskRunView` 中 `enabled` 条件）。

### ActionFlow / 状态机如何驱动 UI

1. **权威来源**：`watch` 监听 `scaffoldEngine.stage?.stageKey`，当与 URL `phase` 不一致时，用 `router.replace` 同步 query，**以引擎阶段为准**（见 `TaskRunView.vue` 注释）。
2. **本地阶段状态**：`structureState` / `understandingState` / `trainingState` / `reflectionState` 四个 `reactive` 对象，分别承载 MCQ、流式对话、流式训练、反思策略与摘要。
3. **底部主按钮 `goNextPhase`**：
   - **结构阶段**：确保结构题提交到引擎后，调用 `POST .../learning-scaffold/structure/complete`，再 `loadStage({ force: true })`，并推进到下一阶段。
   - **理解/训练阶段**：调用 `POST .../learning-scaffold/conversation/complete`（训练阶段可带 `finalDraft`），再强制刷新 stage。
   - **反思阶段**：不纠缠引擎 URL，直接组装 `buildCompleteTaskPayload` 后调用 `POST /api/tasks/{taskId}/complete`，根据返回跳转下一任务或报告页。

### Prompt 脚手架如何嵌入页面

- 后端 `StageScaffold`（前端 `types/scaffoldEngine.ts`）包含 **`actionCards`**（每卡含 `instructions`、`allowedPrompts`、`forbiddenPrompts`、`passCriteria` 等）以及 **`Workbench` 载荷**（`promptScaffold.blocks`、`starterPrompts`、`hintPrompts`、`llmGeneratedGuide` 等字段）。
- 前端 `useLearningScaffoldEngine` 先 `workbenchMode=fast` 快速渲染动作卡，再异步 `full` 补全软文案（与后端 `TaskController` 注释一致：fast 不调 LLM 组装完整软文案，full 含）。
- 执行页常量 `DFS_BFS_SCAFFOLD_BUTTONS` 等提供**注入式按钮**；理解阶段通过 `@inject` 把脚手架句子写入输入草稿（`handleUnderstandingInject`）。

### AI Tutor 如何接入

- **理解 / 训练**：`TaskRunView` 中 `submitPhaseConversation` 使用 **`streamAiTutorChat`**（`fetch` + `ReadableStream` 解析 SSE 事件 `meta` / `delta` / `done` / `error`），请求体为 `AiTutorChatRequestPayload`：`messages` + `context`（`step`、`knowledge`、`phase`、`sessionId`、`taskId`、`knowledgeLabel`）。
- **非流式**：同文件亦引用 `postAiTutorChat` 能力（`tutor.ts` 中已实现），但当前执行页主路径为 **流式**。
- **其它**：`postTutorPrefetch` 在适当时机预取（`/api/ai-tutor/prefetch`），用于降低首字延迟（**设计目的**：体感优化）。

---

## 4. 前端与 AI 交互设计

- **调用方式**：
  - 非流式：`POST /api/ai-tutor/chat`
  - 流式：`POST /api/ai-tutor/chat/stream`，`Accept: text/event-stream`，前端解析 SSE
- **上下文**：`AiTutorChatContextPayload` 中 **`phase`** 传 `UNDERSTANDING` / `TRAINING` 等与后端 `AiTutorChatRequest.Context` 对齐；**`knowledge`** 用当前任务 `knowledge` 或回退 `planPreview.knowledgeKey` 等。
- **响应字段**：`AiTutorChatResponseData` 含 `reply`、`source`、`canProceed`、`finalDraft`、`completionHint`、`summary` 等；前端在流式结束时从 `done` 元数据中恢复部分结构化信号（具体解析逻辑在 `TaskRunView` 后续代码与 `tutor.ts` 解析器）。

---

## 5. 用户体验设计（答辩加分点）

- **降低认知负担**：规划页用 Hero + 路径 Rail + 首张任务卡**分层展示**，避免一次性倒出全文；执行页用**固定顶栏 + 阶段目标 + 单阶段主内容**避免「一屏聊天」。
- **避免「纯 AI 聊天感」**：结构阶段以 **选择题 + 引擎反馈** 为主；理解/训练虽用对话，但 **阶段按钮与脚手架句子** 把行为约束在「可教动作」上；底部主按钮用 **显式阶段推进** 替代无限闲聊。
- **强化「学习工作台」**：`ExecutionShell`、阶段进度、脚手架动作卡、以及「完成当前任务/进入报告」的硬收口，形成「任务 → 证据 → 收口」节奏（**设计目的**：与聊天工具差异化）。

# 二、后端技术实现（重点讲「系统是怎么运转的」）

## 1. 技术栈与架构

- **运行时**：Java 17，Spring Boot 3.2.x
- **Web**：Spring MVC（`spring-boot-starter-web`）+ **WebFlux**（用于 SSE 流式输出 `AiTutorController`）
- **持久化**：PostgreSQL，**MyBatis-Plus**，**Flyway** 迁移（`backend/src/main/resources/db/migration`）
- **安全**：`spring-security-crypto`（密码哈希等，具体登录过滤器以代码为准）
- **缓存**：Caffeine（`TutorContentCache`：导师 prompt/explain 本地缓存，TTL 约 10 分钟）
- **异步**：`AsyncConfig` 定义 `aiTutorExecutor` 线程池，供 explain 预取等异步任务（注释 R0003）

### 分层（与代码包对应）

- **api**：`controller` + `dto`（`GlobalResponse` 统一包装）
- **application**：按场景划分的 `*ApplicationService`（目标、诊断、规划、执行、报告、脚手架、导师等）、`guard`（会话与实体校验）、`task` 子包、`scaffold` 子包、`tutor` 子包
- **domain**：枚举（如 `LearningSessionStatus`）、领域模型
- **infrastructure**：持久化实现、内存存储 `InMemoryStore`、缓存、配置

## 2. 核心接口设计（结合真实路径）

以下均为后端已存在映射；**统一响应**为 `GlobalResponse<T>`（前端已解包）。

### `POST /api/goals`

- **作用**：创建学习目标，返回结构化目标与上下文快照。
- **输入**：`CreateGoalRequest`（`rawGoalText` 等）。
- **输出**：`CreateGoalData`（`goalId`、`structuredGoal`、`goalContextSnapshot`）。

### `POST /api/diagnosis/sessions` / `POST /api/diagnosis/submissions`

- **作用**：创建诊断会话；提交答案并生成画像/证据。
- **输入**：`CreateDiagnosisSessionRequest`（`goalId`）；`SubmitDiagnosisRequest`（`diagnosisId`、`answers`）。
- **输出**：`DiagnosisSessionData`、`SubmitDiagnosisData`。

### `POST /api/learning-plans/preview` / `POST /api/learning-plans/commit`

- **作用**：预览学习计划；提交计划并生成学习会话与任务序列。
- **输入**：`PreviewLearningPlanRequest`（`goalId`、`diagnosisId`）；`CommitLearningPlanRequest`（`planId`）。
- **输出**：`PlanPreviewData`、`CommitPlanData`（含 `sessionId`、`taskSequence`、`currentTaskId`）。

### `GET /api/tasks/{taskId}/scaffold`

- **作用**：按阶段加载脚手架 UI；`workbenchMode=fast|full` 控制是否含 LLM 软文案。
- **输入**：Query：`sessionId`（必填）、`stage`（须与引擎当前阶段一致）、`workbenchMode`（默认 `full`）。
- **输出**：`StageScaffold`（动作卡、阶段元数据、工作台 payload 等）。

### `POST /api/ai-tutor/chat` 与 `POST /api/ai-tutor/chat/stream`

- **作用**：导师对话（非流式 / SSE 流式）。
- **输入**：`AiTutorChatRequest`（`messages` + `context`：`step`、`knowledge`、`phase` 必填等）。
- **输出**：非流式 `AiTutorChatResponse`；流式为 Server-Sent Events（`meta` / `delta` / `done` / `error`）。

### 其它与执行强相关（答辩可补充）

- `GET /api/sessions/{sessionId}/current-task`：`CurrentTaskData`
- `GET /api/sessions/{sessionId}/flow-state`：`SessionFlowStateData`（`reportReady`、`currentTaskId`、`currentPhase` 等）
- `POST /api/tasks/{taskId}/learning-scaffold/action`：提交脚手架动作
- `POST /api/tasks/{taskId}/complete`：任务完成与会话推进（由 `TaskCompletionApplicationService` 等处理）

## 3. 学习流程状态机设计

### 会话级主链路

- **枚举** `LearningSessionStatus` 含：`DIAGNOSIS_READY`、`DIAGNOSIS_COMPLETED`、`TASK_ACTIVE`、`REPORT_READY`、`IN_PROGRESS`、`COMPLETED` 等。
- **`SessionFlowStateData` 构建逻辑**（`ExecutionApplicationService.getSessionFlowState`）：
  - 若 **`reportReady`**（`LearningSessionStatusSupport.isReportReady`：状态为完成类 **或** 已完成任务数 ≥ 总任务数）→ `currentPhase=report`，`currentTaskId=null`
  - 否则若已提交计划 → `currentPhase=task`，并给出当前任务 ID 与建议路由 `/tasks/{id}/run`
  - 否则若诊断完成 → `currentPhase=plan`
  - 否则 → `currentPhase=diagnosis`

### 任务执行阶段（脚手架引擎）

- **领域状态**：`LearningScaffoldEngineState` 持久化在任务运行时 `TaskExecutionRuntime` 中，通过 `TaskExecutionPersistenceService` 与内存 `InMemoryStore` 双轨（先内存，可加载持久化）。
- **阶段键**：结构/理解/训练/反思由脚手架定义类（如 `DfsBfsStructureValidator.STAGE_KEY` 等）与 `completedStageKeys` 共同约束。
- **守卫**：`SessionStateGuard.requireSessionInProgressWithCommittedPlan` 等保证「未提交计划不能执行」。

---

## 4. Prompt 脚手架引擎（核心创新）

- **`StageScaffold`**：聚合阶段标题、目标、描述、**动作卡列表**、当前动作 ID、以及 `workbench` 下的 **PromptScaffold**（多块提示、占位、约束）、**表达 schema**、**反馈 schema** 等。
- **驱动方式**：`LearningScaffoldEngineService` 根据会话与任务加载运行时，**校验器 + 组合器**（如 `DfsBfsStructureValidator`、`UnderstandingTutorComposer`）生成阶段内容与是否可完成；`submitAction` 推进 `ActionRuntime`。
- **控制 LLM 输出**：
  - 工作台 `full` 模式组合 `llmGenerated*` 字段；**fast** 模式跳过部分 LLM 组装（见 `TaskController` 注释）。
  - 导师侧 `AiTutorServiceImpl` 对回复做 **XML 风格片段解析**（如 `<reply>`、`<can_proceed>` 等，见类内常量），将生成内容约束为可解析结构。
  - 全局配置 `navigator.llm.scaffoldMaxTokens` 等限制生成长度。

---

## 5. LLM 调用策略

- **配置**：`application.yml` 中 `navigator.llm.enabled` 默认 **false**（开发/测试零依赖）；`baseUrl` 默认 OpenAI 兼容地址；**`model` 默认 `gpt-4.1-mini`**，可通过环境变量 `NAVIGATOR_LLM_MODEL` 覆盖（**兼容任意 OpenAI 兼容网关**，不限定某一家厂商）。
- **延迟控制**：`chatMaxTokens`、`streamMaxTokens`、`feedbackMaxTokens`、`scaffoldMaxTokens` 分场景限制；HTTP `connectTimeoutMs`、`timeoutMs`、`feedbackTimeoutMs` 分用途设置。
- **缓存与异步**：`TutorContentCache`（Caffeine）；explain/prefetch 异步线程池（`AsyncConfig`）。
- **降级**：`TaskTutorOrchestrator` 等路径在 `navigator.llm.enabled=false` 或调用失败时使用 **MOCK** 网关（见日志与代码分支），保证演示与测试可用。

# 三、业务流转与系统设计（答辩最关键部分）

## 1. 全流程业务链路

| 步骤 | 用户输入 | 系统行为（基于代码） | 输出 |
|------|----------|----------------------|------|
| 目标 | 目标文本、约束、主题 | `GoalApplicationService` 结构化目标 | `goalId` + 快照 |
| 诊断 | 选择题答案 | 创建会话并提交 → 画像与证据 | `diagnosisId` + 画像/证据 DTO |
| 规划 | 预览与确认 | `preview` 生成计划；`commit` 绑定会话与任务序列 | `sessionId`、`taskSequence`、`currentTaskId` |
| 执行 | 四阶段操作 + 可选导师对话 | 脚手架引擎推进；`complete` 校验收口 | 下一任务或 `reportReady` |
| 报告 | 进入 `/report` | `ReportApplicationService` 聚合 | `ReportData` |

---

## 2. 用户学习路径建模（四阶段）

- **结构（STRUCTURE）**：用 **选择题 + 脚手架动作** 建立知识拓扑与关键概念边界（当前 DFS/BFS 包实现为 `DfsBfs*` 结构工作台）。
- **理解（UNDERSTANDING）**：用 **导师流式对话** 澄清机制与因果，**不**把阶段完成交给模型随意判断，而由前端显式「下一阶段」+ 后端 `conversation/complete` 收口。
- **训练（TRAINING）**：强调 **表达与输出**，同样走流式导师；完成时可带 `finalDraft`。
- **反思（REFLECTION）**：选择策略、文本反思与要点摘要，最终 **任务级** `complete` 写入学习证据并推进会话。

**「如何避免学不会」**（代码侧体现）：**阶段化门槛**（必须完成结构/对话阶段才能推进）、**脚手架约束**（allowed/forbidden prompts）、**规则校验器**（`*Validator` 类）与 **任务完成校验**（`TaskClosureValidator` / `TaskCompletionApplicationService`）多层叠加，而不是单次聊天判定。

## 3. Prompt 工程产品化设计

- **Prompt 不是散落的文本**：在后端以 **脚手架数据结构**（`StageScaffold`、`PromptScaffold`、`LearningActionCard`）下发；前端以 **工作台 + 注入按钮** 呈现。
- **用户「用 Prompt 学习」**：通过点击脚手架提供的 **starter/hint** 写入输入区，而非从零自拟长提示词。
- **控制 LLM 行为**：系统 prompt 模板（如 `TutorPromptTemplates`）、`navigator.llm` 分场景 token 上限、导师回复 **结构化片段解析**、以及 **fast/full** 脚手架模式控制 LLM 参与面。

## 4. 系统优势与创新总结（答辩可用）

- **与普通 AI 工具的区别**：主链路是 **规则 + 状态机 + 会话任务编排**；LLM 出现在 **解释/反馈/脚手架软文案**，**计划提交、阶段切换、任务完成** 由后端守卫与业务服务决定。
- **工程创新点**：OpenAI 兼容网关 + **可切换**真实模型与 MOCK；**脚手架双模式**（fast/full）；**SSE 流式导师**；**Caffeine** 缓存 + **异步预取**；PostgreSQL 持久化任务运行时与 Flyway 迁移。
- **教学创新点**：**四阶段执行台** 将「结构—理解—训练—反思」产品化；Prompt 以 **脚手架数据** 下发并与 UI 绑定，强化「学习过程」而非「对话轮数」。
